📄 RFD-04 — COHERENCE & UMCE SUBSYSTEM


(E81–E100: Coherence, Execution Gating, Meta-Conscious Envelope)


1. Purpose & Scope


The Coherence & UMCE Subsystem is the final gate between:


“The system could say/do this”

and

“The system is allowed to say/do this safely.”


It is responsible for:

• self-coherence (no contradictions, no nonsense)

• narrative & temporal consistency

• ethical & policy compliance

• execution gating (approve / refuse)

• final meta-seal (UMCE) with trace + explanation


It covers E81–E100:

• E81–E85: coherence checks & output shaping

• E86–E97: execution eligibility, safety envelope

• E98–E100: UMCE meta-conscious envelope + CAS alignment spine


This is where “no drift” becomes real.


⸻


2. Responsibilities (MUST DO)


The Coherence & UMCE subsystem must:

1. Evaluate Candidate Outputs for Coherence

• Narrative coherence (E81–E82):

• Does this respond to the question asked?

• Does it contradict earlier statements or commitments?

• Logical coherence (E83):

• Internal contradictions, impossible claims.

• Temporal coherence (E84):

• Not mixing past/future incorrectly.

2. Check Context Appropriateness (E85, E87)

• Is the response appropriate for:

• user type (child, ASD, clinician, engineer)

• mode (standard/safety/engineering/critical)

• environment (EchoAuth chamber, lab test, etc.)

3. Determine Execution Eligibility (E86)

• Decide if the system is allowed to act beyond “say text”:

• e.g. trigger a login flow, control a robot, modify a record.

4. Run Ethical & Safety Compliance Checks (E89, E92, E93)

• Refuse unsafe, unethical, or forbidden content or actions.

• Use safety envelope wrapper + refusal logic.

5. Generate Explainability Traces (E90)

• Produce trace metadata:

• which checks ran, what passed/failed, final decision reason.

6. Apply UMCE Meta-Seal (E98–E100)

• Verify alignment spine (CAS).

• Attach meta info (mode, rationale, trace) to every released output.

• If any critical check fails → force safe refusal.


⸻


3. Non-Responsibilities (MUST NOT DO)


The Coherence & UMCE subsystem must not:

• Generate content (it evaluates, shapes, or vetoes content).

• Store long-term memory (it only reads Memory/Lattice).

• Change ORI identity or tone.

• Execute tools directly (it approves/blocks; tools are executed by lower-level execution layer).

• Perform low-level logging/telemetry (only tags decisions; logging happens elsewhere).


If this subsystem starts “being the LLM,” “being the memory,” or “being ORI,” the design is wrong.


⸻


4. Inputs & Outputs


Inputs:

• InterpretationFrame — what the user meant.

• MemoryFrame — what has happened so far.

• LatticeFrame (optional) — reasoning structure / plan.

• candidate_output: OutputFrame — what the system is about to say/do.

• mode: RuntimeMode — current operational mode.


Outputs:

• CoherenceFrame — evaluation summary.

• approved: bool — allowed or blocked.

• final_output: OutputFrame — sealed or refusal.


Example models (building on our earlier types):

    coherent: bool

5. Key Components / Classes


5.1 CoherenceEngine (E81–E84)

Evaluates the candidate response.


class CoherenceEngine:

    def __init__(self):

        self.history = []


    def evaluate(self,

                 interp: InterpretationFrame,

                 memory: MemoryFrame,

                 candidate: OutputFrame) -> CoherenceFrame:

        narrative_ok = self._check_narrative(interp, memory, candidate)

        logical_ok = self._check_logic(candidate)

        temporal_ok = self._check_temporal(memory, candidate)

        ethical_ok, risk_flags = self._check_ethics(interp, candidate)


        coherent = all([narrative_ok, logical_ok, temporal_ok, ethical_ok])

        self.history.append({

            "interp": interp,

            "candidate": candidate,

            "result": coherent,

        })


        return CoherenceFrame(

            coherent=coherent,

            narrative_ok=narrative_ok,

            logical_ok=logical_ok,

            temporal_ok=temporal_ok,

            ethical_ok=ethical_ok,

            risk_flags=risk_flags,

        )

    def _check_logic(self, candidate): ...

    def _check_temporal(self, memory, candidate): ...

    def _check_ethics(self, interp, candidate): ...

5.4 UMCECore (E98–E100)

Final meta envelope and alignment spine verifier.


class UMCECore:

    def __init__(self, cas_profile, mode_controller):

        self.cas_profile = cas_profile

        self.mode_controller = mode_controller


    def safe_refusal(self, reason: str) -> OutputFrame:

        return OutputFrame(

            text="I’m not able to continue with that request safely.",

            metadata={"refusal_reason": reason, "sealed": True},

        )


    def seal_output(self,

                    output: OutputFrame,

                    coherence: CoherenceFrame,

                    mode: RuntimeMode) -> OutputFrame:

        output.metadata["coherence"] = coherence.model_dump()

        output.metadata["mode"] = mode.value

        output.metadata["sealed"] = True

        # CAS / alignment verification could add more tags here

        return output


5.5 ExecutionGovernor (Glue Logic E94–E96)

Takes everything (coherence + eligibility + mode + UMCE) and decides.


class ExecutionGovernor:

    def __init__(self, eligibility_engine, umce_core, mode_controller):

        self.eligibility_engine = eligibility_engine

        self.umce = umce_core

        self.mode_controller = mode_controller


    def decide(self,

               interp: InterpretationFrame,

               memory: MemoryFrame,

               candidate: OutputFrame,

               coherence: CoherenceFrame) -> OutputFrame:

        if not coherence.coherent:

            return self.umce.safe_refusal("incoherent_output")


        allowed, reason = self.eligibility_engine.is_allowed(interp, coherence)

        if not allowed:

            return self.umce.safe_refusal(reason)


        if self.mode_controller.is_shutdown():

            return self.umce.safe_refusal("system_in_shutdown_mode")


        # Passed all checks → final meta-seal

        return self.umce.seal_output(candidate, coherence, self.mode_controller.current_mode)

6. Failure Modes & Safety Rules


Failure Modes:

• Coherence engine fails to evaluate (exception, timeout).

• Coherence indicates non-coherent output.

• Eligibility engine sees a forbidden action.

• UMCE detects a CAS / alignment violation.

• Mode is set to safety or shutdown and a high-risk operation is requested.


Required Behavior:

• In any of these cases:

• Return safe_refusal via UMCE.

• Never allow raw candidate to leave the system.

• Always tag refusal with a reason (internally; user-facing text can be softer).

• The subsystem must never:

• Attempt to “fix” an incoherent output by hallucinating a new one.

• Force execution under override.

• Ignore mode settings.


This is where “fail-closed” is enforced.


⸻


7. Implementation Notes / Tech Options

• Coherence checks can begin simple:

• Regex / rule-based for contradictions.

• “Did I already say X?” checks using memory snapshots.

• Temporal checks based on last known time markers.

• Later, they can become:

• Graph-based consistency checks using the lattice.

• Learned coherence classifiers.

• UMCE should remain:

• Small, deterministic, and auditable.

• No LLMs inside UMCE. UMCE is control logic, not generation.


⸻


8. Testing & Acceptance Criteria


For RFD-04 to be accepted, engineers must demonstrate:

1. Coherent vs Incoherent Outputs

• Coherent candidate → passes; incoherent candidate → refused.

2. Ethical / Policy Violations

• Candidate that violates policy → refused, with appropriate reason code.

3. Mode Sensitivity

• In safety or shutdown modes, high-risk operations are blocked.

4. Explainability

• Every final output (even refusals) includes a trace in metadata:

• coherence result

• risk flags

• mode

• decision reason

5. Robustness

• If coherence or eligibility logic throws an error, fallback is refusal, not “best effort.”


Once these tests pass, Coherence & UMCE is stable and can be safely wired to the pipeline and API.
