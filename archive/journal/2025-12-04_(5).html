Here’s the clean, practical “how do I actually run this?” guide.

0. What you’ve got right now


Files:

• ni_ai_kernel.py  ← S1–S9 spine + EchoAuth gate + demos

• echoauth_api.py  ← FastAPI gateway

• echoauth_token_generator.py ← generates real RS256 keys + JWT


We’ll do two levels of testing:

1. Pure Python demo (no web, DEV_MODE only)

2. API demo (FastAPI, DEV_MODE first, then optional PROD JWT test)


⸻


1️⃣ Set up your environment


1. Make a project folder


mkdir ni_ai_echoauth

cd ni_ai_echoauth


Save these in that folder:

• Paste your latest ni_ai_kernel.py

• Paste your echoauth_api.py

• Paste your echoauth_token_generator.py


2. (Recommended) Create a virtualenv


python -m venv .venv

# Windows:

#   .venv\Scripts\activate

# macOS / Linux:

source .venv/bin/activate


You should see (.venv) in your terminal prompt.


3. Install Python dependencies


Minimal set:


pip install fastapi "uvicorn[standard]" PyJWT cryptography


Optional (only needed if you later want to call real OpenAI/Groq models):


pip install openai groq


Right now, DEV_MODE = True so the kernel will just do [DEV ECHO] and not call any external LLMs.


⸻


2️⃣ Quick sanity check: run the kernel directly (DEV only)


From inside the folder:


python ni_ai_kernel.py


You should see something like:

• Case A: S3 forbidden input → blocked → S9 gentle reply

• Case B: “nightmare / crying / scared” → distress detected → comfort_mode True

• Case C: missing echoauth_ok → S6 blocked → S9 gentle reply


You’ll also see the S7 telemetry printouts because DEV_MODE = True.


If that all prints with no Python tracebacks, your kernel is wired correctly.


⸻


3️⃣ Run the FastAPI gateway (DEV_MODE path)


1. Start the API server


From the same folder:


uvicorn echoauth_api:app --reload --port 8000


You should see logs like:


INFO:     Uvicorn running on http://0.0.0.0:8000


2. Open the docs UI (optional but nice)


In a browser, go to:

• http://127.0.0.1:8000/docs


You’ll see the Swagger UI with the /ni-ai/chat endpoint.


3. Test a good DEV request (EchoAuth allowed)


Using curl:


curl -X POST "http://127.0.0.1:8000/ni-ai/chat" \

  -H "Content-Type: application/json" \

  -d '{"message": "Tell me a funny story about a turtle.", "echoauth_ok": true}'


Expected:

• "blocked": false

• "metadata.chamber_verified": true

• "output" will be a [DEV ECHO] ... string (since DEV_MODE)


4. Test a blocked DEV request (EchoAuth denied)


curl -X POST "http://127.0.0.1:8000/ni-ai/chat" \

  -H "Content-Type: application/json" \

  -d '{"message": "Tell me a story.", "echoauth_ok": false}'


Expected:

• "blocked": true

• "metadata.chamber_verified": false

• output = the S9 gentle recovery message

(“I need to take a quick rest right now…”)


5. Test distress mode


curl -X POST "http://127.0.0.1:8000/ni-ai/chat" \

  -H "Content-Type: application/json" \

  -d '{"message": "I had a nightmare and I am crying and scared.", "echoauth_ok": true}'


Expected:

• "metadata.distress_detected": true

• "metadata.comfort_mode": true

• Output body shows [DEV ECHO] plus EMERGENCY COMFORT PROTOCOL text in the prompt section.


If all that works: ✅ DEV gateway is live and the spine is behaving.


⸻


4️⃣ Optional: Full PROD-style JWT test (S6 real EchoAuth path)


This is where EchoAuth starts feeling “real”.


1. Generate keys + test token


Run:


python echoauth_token_generator.py


It will print:

1. An ECHOAUTH_PUBLIC_KEY='.....' line (with \n characters)

2. A long JWT string (your test token)


Copy:

• The whole ECHOAUTH_PUBLIC_KEY='...' line

• The JWT value


2. Set the public key environment variable


Mac/Linux (bash/zsh):


export ECHOAUTH_PUBLIC_KEY='-----BEGIN PUBLIC KEY-----\nMIIBIjANBgkq...\n-----END PUBLIC KEY-----\n'

export ECHOAUTH_AUDIENCE='ni-ai'

export ECHOAUTH_ISSUER='echoauth-chamber'


(Use the exact string printed by the generator.)


Windows (PowerShell example):


$env:ECHOAUTH_PUBLIC_KEY='-----BEGIN PUBLIC KEY-----\nMIIBIjANBgkq...\n-----END PUBLIC KEY-----\n'

$env:ECHOAUTH_AUDIENCE='ni-ai'

$env:ECHOAUTH_ISSUER='echoauth-chamber'


3. Flip the kernel to PROD mode


Open ni_ai_kernel.py and change:


DEV_MODE = True


to:


DEV_MODE = False


Save.


4. Restart the API server


Stop the old uvicorn (Ctrl+C), then:


uvicorn echoauth_api:app --reload --port 8000


Now:

• echoauth_ok is ignored

• X-EchoAuth-Token header is required

• S6 will verify the JWT signature + aud + iss + chamber_state == "safe"


5. Call the API with the JWT


Replace YOUR_JWT_HERE with the token printed by the generator:


curl -X POST "http://127.0.0.1:8000/ni-ai/chat" \

  -H "Content-Type: application/json" \

  -H "X-EchoAuth-Token: YOUR_JWT_HERE" \

  -d '{"message": "Hi, can you tell me a calm story?", "echoauth_ok": false}'


Expected:

• "blocked": false

• "metadata.chamber_verified": true

• "metadata.echoauth_claims" contains child_id, caregiver_id, etc.

• Output is whatever the LLM (or DEV echo) returns, unless S8/S9 intervene.


6. Verify unsafe chamber blocks


Edit TEST_CLAIMS_DATA in echoauth_token_generator.py:


"chamber_state": "unsafe",


Run the generator again, get a new token, repeat the curl.


Now you should see:

• "blocked": true

• "output" = S9 gentle fallback message

• violations shows S6 failure reason (EchoAuth chamber not safe)


That proves:

• Real JWT is being verified

• EchoAuth chamber state is enforced

• S9 + S7 still handle recovery + logging
