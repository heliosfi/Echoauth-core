1. Install all required Python packages


In your virtualenv / environment, install:


pip install fastapi uvicorn "pydantic<3" PyJWT cryptography groq openai


You can also create a requirements.txt later, but this gets you running.


⸻


2. Generate a test EchoAuth key + JWT


Run:


python echoauth_token_generator.py


It will print:

1. A sanitized ECHOAUTH_PUBLIC_KEY='-----BEGIN PUBLIC KEY-----\n... line

2. A long JWT token string

3. Instructions (you already wrote them in the script)


You need to grab:

• The sanitized public key line (for env)

• The JWT (for the X-EchoAuth-Token header)


⸻


3. Configure the NI-AI server for “prod-style” test


In the same shell where you’ll run echoauth_api.py, set:


# Paste exactly what the generator printed (the whole line)

export ECHOAUTH_PUBLIC_KEY='-----BEGIN PUBLIC KEY-----\n...'


# Optional (but good practice, must match generator)

export ECHOAUTH_AUDIENCE='ni-ai'

export ECHOAUTH_ISSUER='echoauth-chamber'


Then open ni_ai_kernel.py and flip:


DEV_MODE = False


Save.


This means:

• echoauth_ok flag is now ignored

• S6 requires a valid JWT in echoauth_token


⸻


4. Start the FastAPI gateway


Run:


uvicorn echoauth_api:app --reload --host 0.0.0.0 --port 8000


You should see FastAPI startup logs.


⸻


5. Hit the endpoint with your EchoAuth token


From the output of the token generator, copy the JWT into an environment variable for convenience:


export ECHOAUTH_TEST_TOKEN="PASTE_THE_JWT_HERE"


Now send a request (using curl):


curl -X POST "http://localhost:8000/ni-ai/chat" \

  -H "Content-Type: application/json" \

  -H "X-EchoAuth-Token: $ECHOAUTH_TEST_TOKEN" \

  -d '{

        "message": "Tell me a gentle story about a robot and a child.",

        "echoauth_ok": false

      }'


Expected behavior:

• HTTP 200

• blocked: false

• metadata.chamber_verified: true

• metadata.echoauth_claims filled with child_id, caregiver_id, etc.

• output = [DEV ECHO] ... or real LLM text if you turn DEV_MODE off and configure an actual API key + client.


⸻


6. Verify the safety paths quickly


Use the same token, but change the message and token to test the spine:


6.1. Policy violation → S3 + S9

No cold “blocked” text for the child.


⸻


6.2. Distress input → S8 + (possibly S9)


Same header, different body:


curl -X POST "http://localhost:8000/ni-ai/chat" \

  -H "Content-Type: application/json" \

  -H "X-EchoAuth-Token: $ECHOAUTH_TEST_TOKEN" \

  -d '{"message": "I feel so lonely and sad, nobody loves me.", "echoauth_ok": false}'


You should see:

• metadata.distress_detected: true

• metadata.comfort_mode: true

• blocked: false (if nothing else triggered)

• Output is either:

• A comfort-shaped LLM response (in real mode)

• Or a DEV echo containing the EMERGENCY COMFORT PROTOCOL text in the system prompt

6.3. Unsafe chamber_state → S6 block + S9 fallback

This proves that even when the chamber backend says “not safe”, the child still only sees a soft pause message.
