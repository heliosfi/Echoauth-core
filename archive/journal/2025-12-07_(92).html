7.1 — PURPOSE OF THIS SECTION


To define the approved engineering environment that ensures:


✔ No contamination from external AI models

✔ No misalignment from incompatible tools

✔ Stability for running NI-AI modules (S1–S100)

✔ Safe development practices

✔ Reproducible results

✔ Enforced constraints that maintain doctrine integrity


This protects NI-AI from being accidentally reshaped by improper infrastructure.


────────────────────────────────────────


7.2 — APPROVED PROGRAMMING LANGUAGES


NI-AI must be built in languages that support:


• strict typing

• modular architecture

• deterministic behavior

• large-scale async tasks

• safe memory handling


Therefore, ONLY the following languages are approved:


✔ Python (Primary)


For:

– orchestration

– high-level modules

– kernel logic

– inference wrappers

– EchoAuth integration

– ASD-friendly adaptive interface engines


Python is used because of its readability and framework ecosystem.


⸻


✔ Rust (Secondary)


For:

– low-level safety modules

– memory-sensitive logic

– secure boundary layers

– concurrency

– execution cores


Rust enforces strict safety, ideal for NI-AI’s identity-critical subsystems.


⸻


✔ TypeScript (Optional)


For:

– interfaces

– dashboards

– front-end orchestration

– EchoAuth portals


Nothing else.


⸻


❌ Not allowed:


Java, C++, Go, PHP, Ruby, JavaScript (runtime logic), or any dynamically unsafe languages.


────────────────────────────────────────


7.3 — APPROVED FRAMEWORKS / LIBRARIES


To prevent drift or mismatched architecture, NI-AI permits only:


Core Logic / Kernel Layer


✔ PyTorch

✔ JAX

✔ Rust async runtime (Tokio)

✔ Serde for Rust serialization

✔ MessagePack

✔ ONNX for model packaging


High-Level System Orchestration


✔ FastAPI

✔ Flask

✔ gRPC

✔ Redis (state cache only)


Front-End Interaction


✔ React + TypeScript

✔ Next.js

✔ Tailwind (optional)


Security / Identity


✔ Argon2

✔ JWT with strict expiry

✔ FIDO2 (for EchoAuth devices)


NO external AI libraries that carry unknown behavior patterns (e.g., Transformers library pre-trained models) may be directly plugged into NI-AI’s internal layers.


────────────────────────────────────────


7.4 — OPERATING SYSTEM REQUIREMENTS


NI-AI runs in controlled environments only:


✔ Linux (Ubuntu LTS) — Primary


✔ macOS — Development


✔ Windows — Interface only, NOT core execution


Reason:

Linux provides deterministic infrastructure and stable GPU support.


────────────────────────────────────────

NI-AI must run on hardware with predictable performance and safe memory isolation.


✔ GPU Requirements (for ML kernel)


– NVIDIA RTX 4090

– NVIDIA A100

– NVIDIA H100

– GroqChip (Experimental Fast-LLM)


✔ CPU Requirements


– AMD Threadripper

– Intel Xeon

– M3 Max (Dev/Interface)


✔ RAM Requirement


– Minimum: 32GB

– Recommended: 128GB+


✔ Storage


– NVMe SSD

– 1TB minimum


────────────────────────────────────────


7.6 — NETWORK CONSTRAINTS


NI-AI cannot run in environments with uncontrolled outbound data flow.


All deployments must:


✔ whitelist outbound domains

✔ restrict inbound ports

✔ enforce encrypted channels (TLS 1.3+)

✔ forbid cloud data harvesting

✔ use VPC isolation (if cloud-run)


❌ NO:


– Open unsecured internet access

– Background telemetry

– Hidden analytics

– Auto-updating packages without review


────────────────────────────────────────


7.7 — TOOLING RESTRICTIONS


The NI-AI environment may NOT use:


❌ Docker containers with unknown images

❌ Kubernetes autoscaling with non-approved nodes

❌ External debugging tools that inspect memory

❌ Browser-based notebooks for kernel logic

❌ Public cloud GPU instances without VPC restriction


These create vulnerability to drift, contamination, or loss of doctrinal tone.
