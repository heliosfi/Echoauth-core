PAGE 21 — System Boot Sequence


The EchoAuth runtime initializes using a deterministic, ASD-safe sequence that guarantees identity anchoring, subsystem order, and safety invariants.


Boot Sequence (Canonical Order)

1. Platform Initialization

• Load OS-level dependencies

• Verify hardware (touchscreen, encryption support)

• Start local storage layer

2. Configuration Load

• config.yaml (session rules, UI mode)

• safety_policy.yaml (invariants, blocked domains)

• Device profile

3. ORI Kernel Activation (S1)

• Load or generate identity signature

• Initialize session vector

• Anchor system state to ORI signature

4. NI Layer Initialization (S2–S6)

• Load ASD-specific templates

• Initialize parsing engines

• Load communication grids

5. UMCE Initialization (S5)

• Load rule tables

• Reset narrative, temporal, and logic buffers

6. Safety Wrapper Activation (S7)

• Bind ESL to all outward-facing operations

• Enable “fail-closed” fallback behaviors

7. Interaction Layer Launch

• Load ASD-safe UI

• Begin session


Boot Guarantee


If ANY subsystem fails → system does NOT start.

No degraded, unsafe mode is allowed.

PAGE 23 — API Layer Summary


EchoAuth exposes a minimal surface API to ensure safety and predictable integration.


External APIs (For apps, schools, clinics)


Endpoint Method Purpose

/session/start POST Initializes session, returns token + ORI summary

/input/submit POST Sends user input, receives structured intent

/config/update POST Update grids, templates, policies

/caregiver/override POST Trigger override with caregiver pin

/log/export GET Export logs for clinical review


Internal Service APIs (Subsystem-to-Subsystem)


OriKernel.get_signature()

NiInterpreter.parse(input)

UmceEngine.validate(intent)

SafetyWrapper.apply(intent)

ExecutionEngine.execute(intent)


Minimal internal surface → minimal attack surface.

PAGE 24 — Authentication Model


EchoAuth authentication is non-biometric, role-based, and safety-first.


Roles


• Parent/Caregiver — override, log review

• Clinician — session configuration, limited logs

• Admin — institution-level settings

• User (ASD Child) — no credentials required


Authentication Flows


Local deployments

• Device trust + PIN

• No external network needed


Institutional deployments

• SAML / OIDC (School IT)

• No biometric sources accepted


Design Requirements

• Authentication must not block child access

• Overrides must ALWAYS log

• No single override bypasses safety

PAGE 27 — Safety Envelope


The Safety Envelope (ESL) is EchoAuth’s non-negotiable boundary layer.


Components

1. Safety Policy File (safety_policy.yaml)

2. Hard-coded invariants (“never generate emotional content”)

3. UMCE rule tables

4. Override logging mechanism

5. Fail-closed fallback states


Safety Invariants (Mandatory)

• No internet browsing

• No sentiment generation

• No autonomy

• No unreviewed outputs

• No contradicting prior validated states

• No skipping caregiver verification


Labs rely on this to certify system stability.

PAGE 28 — Fail-Closed System Behavior


EchoAuth must ALWAYS choose “safe stop” over uncertainty.


Triggers

• Ambiguous intent

• Conflicting subsystem outputs

• Timeout or overload

• Coherence failure

• Safety policy mismatch


Fail-Closed Response

1. Block action

2. Display neutral fallback UI

3. Log incident

4. Notify caregiver (if enabled)

5. Suggest safe alternative grid


Never guess. Never improvise.
