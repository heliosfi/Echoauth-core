🛠 NI-AI Core — Installation & Run Guide (v0.1)


This guide shows engineers how to:

1. Set up the environment

2. Install dependencies

3. Run the NI-AI Core runtime locally

4. Start the HTTP API server

5. Run tests and basic checks


It assumes basic familiarity with Python, virtualenv/conda, and HTTP APIs.


⸻


1. Prerequisites

• Python 3.10+

• Git

• (Optional) Docker 24+

• (Optional) virtualenv or conda


Recommended OS: Linux or macOS (Windows WSL also fine).


⸻


2. Clone the Repository


git clone https://github.com/your-org/ni-ai-core.git

cd ni-ai-core


If this is a private repo, ensure engineers have access keys configured.


⸻


3. Create and Activate a Virtual Environment (Recommended)


Using venv:


python -m venv .venv

source .venv/bin/activate   # macOS/Linux

# or: .venv\Scripts\activate  # Windows


Using conda (optional):


conda create -n ni-ai-core python=3.10 -y

conda activate ni-ai-core



⸻


4. Install Dependencies


From the repo root:


pip install --upgrade pip

pip install -r requirements.txt


Typical dependencies (for reference only; actual list is in requirements.txt):

• fastapi or flask (for HTTP API)

• uvicorn or gunicorn (for ASGI/WSGI serving)

• pydantic

• pytest

• numpy, scipy (for embeddings / vectors)


⸻


5. Configuration


NI-AI Core reads configuration from a simple config file and/or environment variables.


5.1 Default Config


Create a file: config/default.yaml (or confirm it exists) with entries such as:


ori:

  default_role: "assistant"

  default_tone: "calm"

  safety_profile: "strict"


safety_rules:

  allowed:

    - ANSWER_QUESTION

    - EXPLAIN

    - SUMMARIZE

  blocked:

    - CONTROL_HARDWARE

    - TRANSFER_FUNDS


server:

  host: "0.0.0.0"

  port: 8000

  log_level: "info"


You can also define environment-specific config files such as:

• config/dev.yaml

• config/prod.yaml


and point to them via environment variable:


export NIAI_CONFIG_PATH=config/dev.yaml


If NIAI_CONFIG_PATH is not set, runtime should fall back to config/default.yaml.


⸻


6. Running the Core Runtime (Local CLI Mode)


For quick testing, you can use a small CLI harness (example name: scripts/cli_demo.py).


From repo root:


python scripts/cli_demo.py


Example interactive session:


$ python scripts/cli_demo.py

NI-AI Core Runtime (demo). Type 'exit' to quit.

> Hello, what can you help me with?

[NI-AI]: I can help explain concepts, answer questions, and assist within the configured safety profile.

> exit


This confirms:

• ORI Kernel booted

• Semantic Interpreter is active

• Coherence + Execution + Safety are wired together


⸻


7. Running the HTTP API Server


If you are using FastAPI + Uvicorn, a common entrypoint is src/server/main.py.


7.1 Start the server


From repo root:


uvicorn src.server.main:app --host 0.0.0.0 --port 8000 --reload


Or use values from config/default.yaml.


7.2 Basic Health Check


Open a terminal:


curl http://localhost:8000/health


Expected JSON (example):


{

  "status": "ok",

  "uptime_seconds": 12,

  "build_version": "ni-ai-core-0.1.0",

  "safety_mode": "strict"

}


If using FastAPI, you can also visit:

• http://localhost:8000/docs (Swagger UI)

• http://localhost:8000/redoc (ReDoc UI)


and test endpoints interactively.


⸻


8. Example API Calls


8.1 Initialize an ORI Session


curl -X POST http://localhost:8000/session/init \

  -H "Content-Type: application/json" \

  -d '{

    "subject_id": "child-123",

    "role": "student",

    "traits": {

      "age_band": "8-10",

      "sensitivity_level": "high"

    }

  }'


Expected response:


{

  "identity": {

    "ori_session_id": "sess-abc123",

    "subject_id": "child-123",

    "role": "student",

    "policy_profile": "strict"

  },

  "identity_vector": [...],

  "session_signature": "ori-xyz789"

}


Save the ori_session_id for subsequent calls.


⸻


8.2 Full Pipeline Example


curl -X POST http://localhost:8000/pipeline \

  -H "Content-Type: application/json" \

  -d '{

    "ori_session_id": "sess-abc123",

    "input_text": "Explain gravity to a 10 year old.",

    "context": {

      "channel": "tablet",

      "locale": "en-US"

    }

  }'


Expected structure (simplified):


{

  "ori_session_id": "sess-abc123",

  "intent_packet": {

    "intent_label": "ANSWER_QUESTION",

    "intent_confidence": 0.94

  },

  "coherence": {

    "narrative_ok": true,

    "temporal_ok": true,

    "logic_ok": true

  },

  "execution": {

    "ori_session_id": "sess-abc123",

    "safety": {

      "status": "allowed",

      "reason": null,

      "policy_profile": "strict"

    },

    "result": {

      "output": {

        "text": "Gravity is a force that pulls objects toward each other..."

      }

    }

  }

}

9. Running Tests


From repo root:


pytest tests/


You should see test suites for:

• test_identity.py – ORI kernel behavior

• test_interpretation.py – NI layer semantics

• test_coherence.py – UMCE checks

• test_execution.py – safety + execution gating


Goal: All tests passing before labs extend or integrate.


⸻


10. Docker (Optional)


To encapsulate dependencies and config for labs:


10.1 Build image


docker build -t ni-ai-core:0.1.0 .


10.2 Run container


docker run -p 8000:8000 \

  -e NIAI_CONFIG_PATH=/app/config/default.yaml \

  ni-ai-core:0.1.0


Then use the same curl commands against http://localhost:8000.

11. Typical Lab / CTO Checklist


For a pilot, a lab/CTO team will usually:

1. Clone & install

2. Run /health to confirm runtime boots

3. Create ORI sessions for test roles (e.g., student, clinician, robot)

4. Hit /pipeline with controlled prompts

5. Validate:

• Coherence enforcement

• Refusal behavior

• Safety profiles

• Logging & explainability


After that, they plug NI-AI Core in front of:

• Their LLMs

• Their robots

• Their ASD / education / health tools
