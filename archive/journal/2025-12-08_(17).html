EchoAuth Secure Data Architecture & Regulatory Protection Protocols

SECTION A — DATA SECURITY PRINCIPLES (FOUNDATION)


EchoAuth applies a minimal-data, maximum-safety model designed specifically for neurodivergent populations.

The system stores no biometrics, collects no unnecessary identifiers, and enforces strict access control through the NI-AI Identity Kernel and guardian-verified context signatures.


EchoAuth adheres to five core security principles:

1. Data Minimization

Collect only what is strictly required to support communication, identity confirmation, and safety logic.

2. Local Control First

Whenever possible, data remains on the local device or local school/clinic server.

3. Parent/Guardian Sovereignty

Parents or legal guardians maintain ultimate control of child data, session history, and deletion rights.

4. Role-Based Access Protection (RBAC)

Every data interaction is tied to:

• Who accessed

• Why they accessed

• When they accessed

• What permissions they hold

5. Fail-Closed Security Enforcement

If the system detects mismatched identity, permissions, or context:

all processing stops immediately.


⸻


SECTION B — DATA FLOW (WHAT IS AND IS NOT STORED)


1. What EchoAuth Stores


EchoAuth retains ONLY low-risk, non-sensitive, communication-related metadata:

• session timestamps

• communication choices selected by user

• system state changes (e.g., “break requested”)

• anonymized preference patterns (e.g., low-sensory mode preference)


2. What EchoAuth NEVER Stores


EchoAuth does NOT store:

• facial recognition

• fingerprints or voice prints

• GPS/geolocation

• photos or video

• medical records

• emotional labeling algorithms

• raw voice recordings

• behavioral profiling


These exclusions remove 90% of compliance risk and build trust with parents, schools, and clinicians.

SECTION C — REGULATORY COMPLIANCE FRAMEWORK


EchoAuth is intentionally designed to comply with:


FERPA (Family Educational Rights and Privacy Act)

• No student data is shared without guardian consent

• Schools retain ownership of student educational records

• All data provided by EchoAuth integrates with FERPA-protected IEP workflows


HIPAA (Health Insurance Portability and Accountability Act)


For clinical pilots, EchoAuth:

• stores no Protected Health Information (PHI)

• encrypts all communication-related metadata

• secures clinician accounts through RBAC and the ORI Identity Kernel


COPPA (Children’s Online Privacy Protection Act)


EchoAuth:

• requires guardian consent before activation

• collects only minimal operational data

• contains no social features, messaging, or advertising


The system is structurally aligned with COPPA because it does not track or profile children.


IDEA Part B (Individuals with Disabilities Education Act)


EchoAuth supports:

• IEP communication goals

• data-informed supports

• non-verbal accommodation requirements

SECTION D — DATA PROTECTION MECHANISMS


EchoAuth uses a secure-by-design architecture, including:


1. End-to-End Encryption

• In-transit: TLS 1.3

• At-rest: AES-256


2. Non-Biometric Identity Kernel


Identity is confirmed ONLY through:

• contextual signatures

• session patterns

• pre-authorized guardian/school credentials


3. Guardian-Controlled Access


Guardians can:

• delete data

• export session summaries

• revoke school/clinic access

• disable the system entirely


4. Audit Logging


Every access event is logged:

• role

• timestamp

• session context

• authorization pathway


Audit logs are fully encrypted and never shared externally.


5. Safety Wrapper (UMCE)


The Unified Meta-Control Engine enforces:

• predictable system behavior

• bounded output

• no off-path responses

• immediate shutdown on anomalies


⸻


SECTION E — DATA RETENTION & DELETION POLICY


1. Default Retention


Communication metadata is retained for 30 days, unless guardian requests extension.


2. Guardian-Directed Deletion


Parents can delete:

• single sessions

• entire user histories

• school/clinic access keys

• stored preferences


Deletion is permanent and occurs within 60 seconds.


3. Pilot Program Retention


During research or pilot testing:

• data is anonymized before any analysis

• identifying information is removed

• IRB requirements override default retention ONLY with guardian consent
