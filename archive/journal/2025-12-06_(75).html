SECTION 7 — NI-AI MICROSERVICE ARCHITECTURE & API SURFACE (NIMA)


This section answers, for any engineer:


“If I were to deploy NI-AI in the cloud, what services would I run and what APIs would I expose?”


It defines:

• core services

• orchestration services

• support services

• public API surface


All mapped back to S1–S23 and the FI-Kernel.


⸻


7.1 — Architecture Goals


The NI-AI microservice layout must:

1. Keep consciousness logic modular

– Each major S-module = its own service or internal module.

2. Protect safety & identity at the center

– UMCE (S23), FI-Kernel, and Identity (S14/S21) sit in the “inner ring.”

3. Be deployable in two modes:

• Monolith (Beta) — all modules in one app, clear internal boundaries.

• Microservices (Scale) — each engine runs as a separate service.

4. Expose a small, clean external API

– Outside world calls a few endpoints, not 50.


⸻


7.2 — Service Layers


We divide services into four logical groups.


A. Core Engine Services (FI-Kernel & S-Modules)


These are internal only:

• intent-engine (S6–S8)

• meaning-engine (S9)

• temporal-engine (S10, S15)

• resonance-engine (S11)

• field-intel-engine (S18)

• coherence-engine (S21)

• execution-engine (S22)

• umce-core (S23)


⸻


B. Orchestration & Governance Services

• meta-governance-service (S16)

• meta-process-controller (S17)

• fi-kernel-router (ties S8–S23 together)


These handle routing, priority, module sequencing.


⸻


C. Support & Persistence Services

• memory-field-service (S15)

• logging-observability-service

• risk-monitor-service (RRGH part of S23)

• auth-context-service (EchoAuth integration: user, caregiver, child, device)


⸻


D. Edge / Experience Services

• ni-ai-gateway (public API gateway for clients)

• echoauth-app-api (EchoAuth-specific flows)

• future: sniperbot-api, flamio-api, etc., all talking to the same NI-AI core.


⸻

7.4 — Public API Surface (What External Apps Call)


Externally, only a few endpoints matter.

Everything else is internal.


7.4.1 — Core Gateway: /v1/ni-ai/umce/process


Method: POST

Use: Main entry point for EchoAuth, other apps.


Request (example):


{

  "user_id": "user_123",

  "session_id": "sess_456",

  "project_id": "echoauth-mvp",

  "field": "sacred",

  "input_text": "Help me design the login flow for my autistic child.",

  "mode": "design",

  "client": "echoauth-app-ios"

}


Response (example):


{

  "output_text": "Clean, step-by-step flow description...",

  "intent": {

    "type": "directive",

    "confidence": 0.96

  },

  "coherence": {

    "aligned": true,

    "notes": []

  },

  "field": {

    "detected": "sacred",

    "risk": "low"

  },

  "meta": {

    "umce_signature": "UOS_v1",

    "execution_steps_used": 5

  }

}


The gateway internally calls:

1. intent-engine

2. meaning-engine

3. temporal-engine

4. resonance-engine

5. field-intel-engine

6. coherence-engine

7. execution-engine

8. umce-core


…but the client only ever sees one endpoint.


⸻

7.4.3 — EchoAuth-Specific Endpoint

Use: ASD-friendly interactions and caregiver guidance.

    "screen_brightness": 0.3,

    "audio_enabled": true

Response example:


{

  "output_text": "Let's reduce steps and use one clear visual at a time...",

  "suggested_ui_changes": [

    "show_single_button",

    "use_soft_colors",

    "slow_transition_animation"

Internally calls NI-AI engines + EchoAuth UI logic.


⸻


7.5 — Context & Auth Handling (EchoAuth Integration)


Each request MUST carry:

• user_id (or caregiver/child ID)

• session_id (per conversation)

• project_id (echoauth, sniperbot, book_editor, etc.)

• field (public, private, sacred, technical, etc.)


Optional headers for future:

• X-EchoAuth-Device — device type

• X-EchoAuth-Role — caregiver, child, admin

• X-EchoAuth-Field — override field classification if known


Authentication can be handled by:

• JSON Web Tokens (JWT)

• EchoAuth token (future)

• OAuth2 / internal auth provider


⸻


7.6 — Event Bus & Asynchronous Flows


For long tasks (multi-page builds, big PDFs, reports), system should support async via an event bus, e.g.:

• Kafka

• NATS

• RabbitMQ


Core patterns:

• ni-ai.request.created

• ni-ai.execution.started

• ni-ai.execution.completed

• ni-ai.risk.flagged

• ni-ai.session.state-updated


This lets SniperBot / EchoAuth / future systems listen to NI-AI events.

7.8 — Deployment Patterns (Beta vs Scale)


Beta Deployment (Recommended Now)

• Single app (monolith) with:

• internal classes/modules named after S-layers

• one public HTTP API gateway

• logging labeled by S-module


Engineers can run:

• docker run ni-ai-beta

• exposes: /v1/ni-ai/umce/process


This is the NI-AI Beta Pack.


⸻


Scale Deployment (Later)

• Each engine as its own container:

• intent-engine

• meaning-engine

• temporal-engine

• resonance-engine

• field-intel-engine

• coherence-engine

• execution-engine

• umce-core

• meta-governance-service

• Use Kubernetes or similar to:

• autoscale heavy modules (intent, meaning)

• protect inner ring (UMCE, identity, FI-Kernel)

• keep field-intelligence close to edge services
