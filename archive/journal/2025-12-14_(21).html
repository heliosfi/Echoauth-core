STAGE 3 — SYSTEM SECURITY PLAN (SSP)


NI-AI × EchoAuth × MCG

(Federal-Aligned, Governance-First)


⸻


3.1 Purpose of the SSP


This System Security Plan documents how NI-AI, EchoAuth, and MCG protect:

• users (especially non-verbal children),

• caregivers and institutions,

• system integrity,

• and government stakeholders,


through preventive controls, fail-closed behavior, and auditability.


The SSP demonstrates alignment with:

• NIST 800-53 Rev. 5

• FISMA

• Federal Zero Trust principles


This system is designed to be secure by restraint, not by expansion.


⸻


3.2 System Scope & Boundary


In-Scope Components

• MCG (Governance Spine)

Final authority for allow / pause / deny.

• EchoAuth (Authorization Layer)

Role-based access and identity verification.

• NI-AI (Explanation Layer)

Read-only transparency and decision explanation.

• Logging & Audit Subsystem

• Configuration & Policy Store


Out-of-Scope

• End-user devices

• External networks

• Third-party systems (unless explicitly integrated)

• Any autonomous actuation outside human approval


Boundary rule: Nothing executes outside MCG authority.


⸻


3.3 Security Architecture Overview


Security is layered and non-bypassable:

1. Perimeter Security

• Encrypted transport (TLS 1.3)

• Network segmentation

2. Identity & Access Control

• EchoAuth enforces role-based access

• Default state = deny

3. Governance Enforcement

• MCG evaluates state before any action

• Uncertainty resolves to pause

4. Explanation & Oversight

• NI-AI explains outcomes

• No hidden logic paths

5. Audit & Logging

• Immutable, reviewable records

• No raw inference storage


⸻


3.4 Access Control (NIST AC Family)

• AC-2 Account Management

Only authorized roles (caregiver, admin, reviewer).

• AC-3 Access Enforcement

Role + context required.

• AC-6 Least Privilege

Minimum permissions at all times.

• AC-7 Session Control

Inactivity collapses to observation.


⸻


3.5 Identification & Authentication (IA)

• IA-2 Multi-Factor Authentication

EchoAuth handles identity and role verification.

• IA-5 Authenticator Management

Credentials are managed, rotated, revocable.

• IA-8 Non-Person Authentication

Device/session identifiers where applicable.


No child is required to authenticate independently.


⸻


3.6 System & Communications Protection (SC)

• SC-7 Boundary Protection

No lateral movement between layers.

• SC-12 / SC-13 Cryptographic Protection

AES-256 at rest, TLS in transit.

• SC-28 Protection of Data at Rest


⸻


3.7 System Integrity (SI)

• SI-4 Continuous Monitoring

State validation and drift detection.

• SI-7 Integrity Checks

Configuration and policy enforcement.

• SI-10 Error Handling

Errors trigger pause, not retries.

• SI-11 (Custom)

Non-action under uncertainty is enforced.


⸻


3.8 Incident Response (IR)


If anomalies occur:

1. Detect (state violation or instability)

2. Contain (pause execution)

3. Preserve logs

4. Notify authorized personnel

5. Resume only after review


No automated escalation involving the child.


⸻


3.9 Logging & Audit (AU)

• All allow / deny / pause decisions logged

• Time-stamped

• Immutable

• Reviewable by authorized reviewers


Logs are used for accountability, not surveillance.


⸻


3.10 Security Posture Summary

• Fail-closed by default

• Governance precedes capability

• No autonomy

• No inference as truth

• Human oversight is mandatory
