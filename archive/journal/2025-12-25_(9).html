SNIPERBOT — v1.1 CHANGE RULES (CANONICAL)


Purpose


v1.1 exists to improve discipline and robustness, not to chase performance or novelty.


v1.0 remains frozen and authoritative.


⸻


1) What v1.1 IS Allowed to Change


Only changes that improve safety, clarity, or enforcement are permitted.


✅ Allowed Categories

• Gate thresholds

• Volatility limits

• Spread caps

• Delta / theta ranges

• Time parameters

• Card expiration windows

• Time stops

• Logging & audit enhancements

• Additional fields

• Clearer failure reasons

• UX / operator clarity

• Better alerts

• Clearer state visibility

• One additional whitelisted setup

• Only if it is structurally defined

• Must be simpler than existing ones


All allowed changes must:

• Reduce ambiguity

• Reduce discretionary behavior

• Increase consistency


⸻


2) What v1.1 is NEVER Allowed to Change


These are hard invariants.


❌ Forbidden Changes

• Default state ≠ PAUSE

• Removal of Trade Card requirement

• Allowing execution without stop + TP

• Increasing risk per trade above v1.0 limits

• Removing LOCKOUT behavior

• Allowing continuous auto-trading

• Allowing prediction-based entries

• Allowing AI to override human authority

• Making Auto-Mode the default


If any of these change → it is not SniperBot.


⸻


3) Change Proposal Format (Required)


Every v1.1 change must be written before implementation:


Change Proposal Template

• Change name

• Version: v1.1.x

• What is changing (exact)

• Why (discipline/safety reason)

• What does NOT change

• Expected behavioral effect

• Risk of unintended consequence


No written proposal → no change.


⸻


4) Testing Requirement (Mandatory)


Every v1.1 change must pass:


Phase A — PAPER

• Minimum 5 full sessions

• No live capital

• All gates active


Phase B — HYBRID

• Minimum 5 sessions

• Reduced risk (≤ 0.25%)

• Manual execution only


If discipline degrades at any point → change rejected.


⸻


5) Graduation Rules


A change may graduate only if:

• Rule violations decrease or remain zero

• “No Trade” days increase or stay stable

• Emotional neutrality improves

• Execution feels more boring, not exciting


Profit alone is not a pass condition.


⸻


6) Version Control Discipline

• v1.0 remains untouched

• v1.1 changes are additive or parametric only

• Each accepted change increments v1.1.x

• Rejected changes are archived, not deleted


⸻


7) Emergency Rollback Rule


At any time:

• Revert instantly to v1.0

• No justification required

• Rollback is governance, not failure


⸻


8) The One-Sentence Lock


v1.1 may only make SniperBot harder to misuse — never easier to trade.
