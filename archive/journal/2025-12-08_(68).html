📄 RFD-03 — MEMORY & GOVERNANCE FLOW SUBSYSTEM


(E36–E60: MFAL, sessions, continuity, stability, flow)


1. Purpose & Scope


The Memory System is responsible for:

• holding session state

• maintaining continuity of thought

• managing short- and long-range context

• supporting predictive stability and state reconciliation

• providing a safe, controlled interface for all read/write operations


It covers E36–E60, and is used by:

• ORI (identity/continuity)

• Semantic Interpreter (context, user model)

• Lattice / UCS (longer-term reasoning)

• Coherence / UMCE (self-consistency checks)

• Execution / Safety (knowing what already happened)


This subsystem is not optional. If Memory is misdesigned, NI-AI will drift, repeat, contradict, or lose the user.


⸻


2. Responsibilities (MUST DO)


The Memory System must:

1. Provide a Memory Field Abstraction Layer (MFAL)

• E36: unify access to all memory types (session, project, long-term).

• Present a single interface for reading/writing state.

2. Manage Session Memory

• E37–E40: track per-session histories and signatures.

• Store recent interactions, key flags, active tasks.

3. Handle Temporal Continuity

• E38: carry relevant state across turns and (optionally) across sessions.

• E56: manage long-range tasks and continuity markers.

4. Maintain Memory Integrity

• E41–E43: validate, correct drift, and reconcile conflicting states.

5. Bind Memory & Coherence

• E45: expose memory hooks so Coherence Engine can:

• check narrative consistency

• compare current output against past commitments

6. Govern Flow & Load

• E48–E52: route requests, prioritize important tasks, prevent overload.

• E54–E55: handle interruptions and safe recovery.

7. Expose a Clear API

• Back /session/start, /memory/store, /memory/get, /coherence/validate.


⸻


3. Non-Responsibilities (MUST NOT DO)


The Memory System must not:

• Generate natural language responses.

• Apply ethical or safety judgments (that’s Coherence/UMCE).

• Change ORI identity vectors.

• Act as an LLM or inference engine.

• Store unbounded raw logs without retention rules.


If memory begins “deciding,” “responding,” or “overriding identity,” the design is wrong.


⸻


4. Inputs & Outputs


Core internal inputs:

• InputFrame (for logging and context)

• InterpretationFrame (for structured semantic state)

• OutputFrame (for storing decisions/outcomes)

• session_signature (from ORI)


Core internal outputs:

• MemoryFrame — the consolidated memory snapshot for downstream components

• history_slice — a filtered set of past turns relevant to current query

• task_state — representation of ongoing tasks, goals, flags (e.g. “login flow in progress”)


Example structure:


class MemoryFrame(BaseModel):

    session_id: str

    history: List[Dict[str, Any]]        # condensed entries, not full logs

    active_goals: List[str]

    last_updated: float

    stability_flags: List[str]           # e.g. ["stable", "reconciled"]

6. Failure Modes & Safety Rules


Potential failure modes:

1. Memory Overflow / Bloat

• Unlimited history growth → latency, instability.

2. Corrupted State

• Partial writes, inconsistent flags, broken JSON, etc.

3. Contradictory State

• Memory says “task complete” while current flow behaves as if it’s not.

4. Orphaned Sessions

• No matching session for active requests.

5. Leaking Sensitive Data

• Storing PII or unnecessary sensitive context.


Required safety behaviors:

• E54 (Overload Prevention)

• Trim/prune history using FlowCoordinator.trim_history().

• Keep only recent window + key markers (goals, decisions, commitments).

• E42 (Memory Integrity Validator)

• Run integrity checks periodically.

• On failure, mark stability_flags and notify UMCE.

• E43 (Memory Drift Corrector)

• When contradictions are detected, reconcile or fallback to safest interpretation, e.g.:

• prefer “task not complete” over “complete” to avoid skipping safety steps.

• On any critical memory error:

• Fail closed: return a safe refusal or ask to restart the flow.

• Do not fabricate continuity.

7. Implementation Notes / Tech Options

• Backend

• Prototype: in-memory (Python dict), suitable for single-process demos.

• Later: Redis, Postgres, or similar with TTL and retention policies.

• Retention

• Session history truncated to N turns (e.g., last 20 exchanges) + key markers.

• Old sessions auto-expired after a configurable period.

• Schema

• Use MemoryFrame as the internal canonical structure.

• Avoid storing raw LLM tokens; store semantic summaries.

• Privacy / Compliance

• Do not store full PII unless required.

• Redact or anonymize when possible (especially for ASD / child use).

8. Testing & Acceptance Criteria


For RFD-03 to be considered implemented and accepted, engineers must provide automated tests that prove:

1. Session Creation & Retrieval

• Creating a session → returns MemoryFrame.

• Reloading returns the same session_id and stable active_goals.

2. History Append & Trim

• Appending events grows history.

• When history length > max, trimming keeps recent events and required markers.

3. Integrity & Reconciliation

• Inject corrupted or contradictory states → StabilityManager flags them.

• Reconciliation either resolves or marks them as “requires intervention”.

4. Overload Prevention

• Under heavy event load, system stays responsive and does not exceed configured memory limits.

5. Flow Recovery

• Interruption & resume markers work: system can restore context after a break.

6. Security / Privacy

• Tests confirm that disallowed fields (e.g. raw passwords) are not stored.


Once these pass, Memory System is safe to integrate with Coherence, Execution, and UMCE.
