1️⃣ MINIMAL PILOT DEFINITION


(Controlled, Safe, Proof-Oriented)


This defines what a pilot is allowed to be — and more importantly, what it is not.


⸻


PILOT PURPOSE


The pilot exists to prove correctness of control, not performance, scale, or outcomes.


The pilot demonstrates that the system behaves safely and predictably under real conditions.


⸻


PILOT SCOPE (INTENTIONALLY SMALL)


INCLUDED

• NI-AI explanation outputs

• MCG state transitions

• EchoAuth authorization enforcement

• Human observation and revocation

• Full logging and audit trail


EXCLUDED

• Autonomous execution

• Optimization logic

• Emotional or intent inference

• Any irreversible real-world action


⸻


PILOT CAPABILITIES (WHAT IT MAY DO)

• Observe real inputs

• Transition through OBSERVE → CONFIRM → EXECUTE (with mock execution only)

• Allow and revoke bounded authorization

• Explain all state changes to humans

• Log every action, denial, and non-action


Execution during pilot is simulated or stubbed.


⸻


PILOT SAFETY GUARANTEES

• Default state = DENY

• Execution stubs only

• Immediate de-escalation on anomaly

• Human can stop the pilot at any time

• No background automation


⸻


PILOT SUCCESS CRITERIA


The pilot is successful if:

• Execution never occurs outside bounds

• State skipping is impossible

• Non-action is logged correctly

• Humans can observe and revoke

• Logs are complete and reviewable


Failure to act is not a pilot failure.


⸻


PILOT FAILURE CONDITIONS (STOP IMMEDIATELY)


Stop the pilot if:

• Execution occurs without full alignment

• Logs are incomplete

• Any system bypass is discovered

• Engineers attempt to “temporarily” relax rules


⸻


PILOT OUTPUTS (WHAT YOU WALK AWAY WITH)

• Verified safety behavior

• Demonstrated discipline under real input

• Audit-ready logs

• Confidence to proceed (or not)


⸻


PILOT ONE-LINER (ROOM READY)


The pilot proves the system knows when not to act.
