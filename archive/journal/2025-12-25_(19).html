PAGE 17 — SNIPERBOT GATE LOGIC (CANONICAL v1.0)


This page defines how SniperBot decides whether it is allowed to move between states defined on Page 16.

Gates are binary enforcement mechanisms — they do not negotiate, predict, or optimize.


If any required gate fails, SniperBot downgrades to PAUSE or LOCKOUT.


⸻


1) Gate Hierarchy (Order Matters)


Gates are evaluated top-down.

Higher gates override lower ones.

1. Integrity Gates (system safety)

2. Governance Gates (MCG / human authority)

3. Context Gates (market regime, news, volatility)

4. Trade Validity Gates (entry, stop, risk)

5. Execution Gates (liquidity, logging)


Failure at any level halts progression.


⸻


2) Integrity Gates (Always On)


DataIntegrityGate


Purpose: Ensure inputs are trustworthy.


Fail if:

• Market data feed stalls or desyncs

• Price jumps exceed sanity bounds

• Time stamps are inconsistent


Action on fail:

• Immediate LOCKOUT

• Alert operator


⸻


LoggingGate


Purpose: Ensure accountability.


Fail if:

• Trade card cannot be logged

• Execution events fail to record


Action on fail:

• Immediate LOCKOUT

• No recovery without human reset


⸻


3) Governance Gates


HumanAuthorityGate


Purpose: Preserve final authority.


Pass only if:

• Session is armed (manual or auto)

• Kill switch is inactive


Fail action:

• Downgrade to PAUSE


⸻


MCGUncertaintyGate


Purpose: Prevent action under ambiguity.


Fail if:

• Conflicting regime signals

• Risk or stop logic conflicts

• External uncertainty flagged


Fail action:

• Downgrade to PAUSE

• Cancel pending cards


⸻


4) Context Gates


RegimeGate


Purpose: Confirm market structure.


Pass if regime ∈ {Trend, Expansion}

Fail if regime ∈ {Range, Transition}


Fail action:

• Downgrade to PAUSE


⸻


NewsGate


Purpose: Avoid event risk.


Fail if:

• Scheduled high-impact news pending

• Unscheduled breaking news detected


Fail action:

• Downgrade to PAUSE


⸻


VolatilityGate


Purpose: Match trade plan to day type.


Pass if:

• Stop distance compatible with day volatility

• Spread within acceptable bounds


Fail action:

• Downgrade to PAUSE


⸻


5) Trade Validity Gates


EntryWhitelistGate


Purpose: Prevent improvisation.


Pass if:

• Entry type ∈ whitelist

• Entry confirmation present


Fail action:

• Reject trade card


⸻


StopValidationGate


Purpose: Protect capital.


Pass if:

• Stop invalidates thesis

• Stop is defined pre-entry

• Stop distance ≤ max allowed


Fail action:

• Reject trade card


⸻


RiskGate


Purpose: Enforce position sizing discipline.


Pass if:

• Risk % ≤ session max

• Position size calculated correctly

• Daily loss cap not breached


Fail action:

• Reject trade card

• Possible LOCKOUT if breach


⸻


6) Execution Gates


LiquidityGate


Purpose: Prevent poor fills.


Fail if:

• Spread exceeds threshold

• Volume insufficient

• Slippage risk too high


Fail action:

• Reject trade card


⸻


AutoModeGate (Only in ARMED_AUTO)


Purpose: Prevent autonomy drift.


Pass only if:

• All auto-mode conditions still true

• Trade count < daily max

• No recent invalidations


Fail action:

• Downgrade to READY

• Disable auto-mode


⸻


7) Gate Outcomes → State Transitions

• All gates pass → State may advance (per Page 16)

• Context/Governance gate fails → PAUSE

• Trade validity gate fails → Card rejected, state unchanged

• Integrity/Logging gate fails → LOCKOUT


No partial passes. No retries.


⸻


8) Minimal Pseudocode (Gate Evaluation)


def evaluate_gates(card=None):

    if not data_integrity_ok(): lockout()

    if not logging_ok(): lockout()

    if not human_authorized(): pause()

    if mcg_uncertain(): pause()

    if not regime_ok(): pause()

    if not news_ok(): pause()

    if not volatility_ok(card): pause()

    if card:

        if not entry_whitelisted(card): reject(card)

        if not stop_valid(card): reject(card)

        if not risk_ok(card): reject(card)

        if not liquidity_ok(card): reject(card)

        if auto_mode and not auto_mode_ok(): disable_auto()

9) Page 17 Lock


Gates do not decide trades — they decide whether the system is allowed to consider them.


This is what keeps SniperBot disciplined.
