📄 RFD-02 — Semantic Interpreter (NI Layer)


1. Purpose & Scope


The Semantic Interpreter is the NI layer that turns raw user input into structured meaning:

• parses text

• extracts intent(s)

• tags temporal info (past / now / future)

• reads emotional tone (detection only)

• prepares a safe, structured packet for memory, lattice, and coherence


Scope:

Covers E16–E35 and connects ORI → Memory → Coherence.


⸻


2. Responsibilities (MUST DO)


The Semantic Interpreter must:

1. Normalize Input

• Clean text (whitespace, encoding, basic normalization).

• Handle simple multilingual cases if configured.

2. Parse & Tokenize

• Break text into units suitable for intent/semantic models.

3. Extract Intent

• Determine primary intent (e.g. “ask_question”, “request_explanation”, “give_feedback”).

• Optionally identify secondary intents.

4. Build Meaning Representation

• Produce a meaning_graph or equivalent:

• key entities, actions, objects, and relationships.

5. Temporal Tagging

• Tag references to time (now / future / past event).

6. Emotional Tone Detection (No Generation)

• Detect emotional tone (e.g. calm, frustrated, confused).

• Never synthesize emotion; only label it.

7. Confidence & Risk

• Compute a confidence score.

• Flag low-confidence interpretations for UMCE or safer output.

8. Governance Hooks

• Apply Semantic Risk Filter (E29) to flag harmful or ambiguous content.


⸻


3. Non-Responsibilities (MUST NOT DO)


The Semantic Interpreter must not:

• Generate any natural language responses.

• Decide what the system will ultimately do or say.

• Modify identity or ORI constraints.

• Store long-term history (beyond what’s needed for immediate interpretation).

• Perform ethical judgment beyond risk tagging (coherence & UMCE handle that).


If interpretive logic starts “deciding” or “responding,” it’s outside this RFD.


⸻


4. Inputs & Outputs


Input: from API / ORI


class InputFrame(BaseModel):

    user_id: str

    text: str

    timestamp: float

    metadata: Dict[str, Any] = {}


Output: InterpretationFrame


class InterpretationFrame(BaseModel):

    parsed: Dict[str, Any]            # raw parse / tokens / structure

    intent: str                       # primary intent

    intents_secondary: List[str]      # optional secondary intents

    meaning_graph: Dict[str, Any]     # entities, relations, etc.

    temporal_tags: Dict[str, Any]     # time references

    emotional_read: Dict[str, float]  # e.g. {"calm": 0.8, "angry": 0.1}

    confidence: float                 # 0.0–1.0

    risk_flags: List[str]             # e.g. ["ambiguous", "self_harm_hint"]

    identity: Any                     # ori_kernel.identity_vector

    session: Any                      # ori_kernel.session_signature


This InterpretationFrame is what Memory, Lattice, and Coherence will use.

6. Failure Modes & Safety Rules


Failure Modes:

• Input text is empty or non-linguistic.

• Models return low confidence or conflicting intents.

• Emotional signal suggests high distress but content is unclear.

• Parse fails due to encoding / language issues.


Required Behavior:

• On parsing/intent failure:

• Set intent="unknown", confidence low, add risk_flags=["low_confidence"].

• On high emotional distress:

• Add risk_flags=["distress_signal"] for UMCE to act on.

• Never fabricate high confidence; uncertainty must be explicit.

• Never override ORI; if ORI marks input as forbidden, interpreter must not proceed.


⸻


7. Implementation Notes / Tech Options

• Early version:

• Simple regex / keyword + small classifier for intents.

• Lightweight emotion classifier (even heuristic).

• Later version:

• Fine-tuned LLM or transformer for intent + meaning graph.

• Cross-lingual support.


All implementations must maintain:

• explicit confidence

• clear risk_flags

• no emotional generation


⸻


8. Testing & Acceptance Criteria


Engineers must provide tests that verify:

1. Basic Parsing

• Normal sentences produce non-empty parsed.

2. Intent Accuracy (Baseline)

• Given a labeled test set, primary intent matches expected label above threshold.

3. Risk Flagging

• Ambiguous or adversarial inputs lead to risk_flags being set.

4. Distress Detection

• Inputs with clear emotional distress produce expected emotional_read pattern and risk flags.

5. Robustness

• Non-ASCII, emojis, minor noise do not crash the system.

6. Integration

• InterpretationFrame passes cleanly into Memory and Coherence components in the starter pipeline.


Once these pass, RFD-02 is considered “implementable and stable.”
