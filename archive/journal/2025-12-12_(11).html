SECTION V — DEVELOPER INTEGRATION GUIDE


5.1 Purpose of This Guide


This section defines how developers integrate, extend, and deploy EchoAuth while preserving:

• Caregiver authority

• Child safety

• Governance (MCG)

• NI-AI constraints


Any implementation that conflicts with this guide is out of scope for the MVP.


⸻


5.2 Integration Principles (Non-Negotiable)


Developers must adhere to the following principles:


1) Human-in-the-loop only

2) No autonomous system decisions

3) No inference of internal child states

4) All actions must be auditable

5) Governance executes before persistence


EchoAuth is support software, not decision software.


⸻


5.3 Authentication & Access Integration


Required

• Session or JWT authentication

• RBAC enforced at API and UI layers


Roles


Caregiver → create/view observations

Parent    → view observations

Admin     → manage users, review logs


Prohibited


❌ Child authentication

❌ Shared or anonymous accounts

❌ Role escalation without admin approval



⸻


5.4 API Design Rules


Allowed API Operations


POST   /observations

GET    /observations

GET    /audit-log

POST   /exports


Disallowed API Operations


❌ /predict

❌ /analyze

❌ /recommend

❌ /classify

❌ /diagnose


If an endpoint suggests interpretation, it violates scope.


⸻


5.5 MCG Middleware Integration


MCG must be implemented as pre-request middleware.


Required Middleware Checks (in order):

1. Identity Anchoring

• Validate authenticated adult user

• Verify role authorization

2. Action Validation

• Confirm action is permitted for role

• Reject disallowed operations

3. Safety Gate

• Block clinical, diagnostic, or predictive actions

4. Audit Logging

• Log action as allowed or blocked

• Store timestamp and user ID


No request bypasses MCG.


⸻


5.6 UI Integration Rules


Caregiver Dashboard

• Manual observation entry only

• Free-text notes allowed

• Historical timeline view

• Export controls (CSV / PDF)


Child-Facing UI (Optional)

• Display-only

• No input handling

• No system-triggered prompts

• No adaptive behavior


UI Prohibitions


❌ Emotion labels

❌ Behavioral scores

❌ Progress ratings

❌ Recommendations



⸻


5.7 Data Handling Requirements


Storage

• PostgreSQL (relational)

• Audit logs in append-only table


Data Constraints


- No biometric inference fields

- No emotional state fields

- No cognitive assessment fields


Retention

• Configurable retention window

• Admin-controlled deletion policies


⸻


5.8 Export & Review Integration


Exports must:

• Be human-readable

• Contain raw observations only

• Include timestamps and caregiver attribution


Allowed formats:


- CSV

- PDF


Exports must never include system interpretation.


⸻


5.9 Testing & Validation


Required Tests


✔ Role-based access tests

✔ Governance block tests

✔ Audit log integrity tests

✔ Export accuracy tests


Prohibited Tests


❌ Accuracy of interpretation

❌ Prediction performance

❌ Behavioral classification


Testing validates safety and compliance, not intelligence.


⸻


5.10 Extension Policy


Any future feature must:

1. Be caregiver-initiated

2. Pass MCG governance checks

3. Avoid interpretation or diagnosis

4. Be auditable


Features requiring inference or prediction are explicitly excluded from MVP and pilot phases.


⸻


5.11 Developer Acknowledgment (Recommended)


All engineers integrating EchoAuth should acknowledge:


“EchoAuth is a caregiver-guided, non-clinical system.

All meaning, interpretation, and decisions remain human-driven.”


This protects the system and the team.
