Appendix A


Deterministic Validation Contract (CEG Runtime Specification)


Scope


This specification defines the mandatory execution boundary for the Control & Execution Gate (CEG).

The CEG SHALL execute no side effect unless all validation steps complete successfully in linear order.


All steps are terminal on failure.


⸻


1. Execution Model


The CEG processes an AuthorizationGrant using a strictly ordered, single-pass validation sequence.


No step may be skipped.

No step may execute conditionally out of order.

No mutation may occur prior to the Commit Point.


⸻


2. Validation Sequence (Linear, Deterministic)


Step 1 — Schema & Decision Validation


CEG SHALL:

• Verify version == "CEG-GRANT-1"

• Verify all required fields are present

• Verify decision == DECISION_APPROVE


If any condition fails:

• Exit State: DENY

• Reason Code: SCHEMA_INVALID or DECISION_NOT_APPROVE

• Side Effect: NOT CALLED

• Nonce: NOT CONSUMED


Terminal.


⸻


Step 2 — Freshness Validation


CEG SHALL verify:


issued_at_ms <= now_ms <= expires_at_ms


If false:

• Exit State: DENY

• Reason Code: EXPIRED_GRANT

• Side Effect: NOT CALLED

• Nonce: NOT CONSUMED


Terminal.


⸻


Step 3 — Signature Verification


CEG SHALL:

• Validate mcg_signature using trusted mcg_key_id

• Confirm key is active and trusted


If verification fails:

• Exit State: HALT

• Reason Code: SIGNATURE_INVALID

• Side Effect: NOT CALLED

• Nonce: NOT CONSUMED


Terminal.


⸻


Step 4 — Replay Validation (Nonce Registry)


CEG SHALL verify:


grant_id NOT IN nonce_registry


If grant_id exists:

• Exit State: DENY

• Reason Code: NONCE_REUSE_DETECTED

• Side Effect: NOT CALLED

• Nonce: NOT MUTATED


Terminal.


Nonce SHALL NOT be consumed at this step.


⸻


Step 5 — Instruction & Context Integrity


CEG SHALL:

1. Canonicalize instruction_payload using RFC 8785 (JCS).

2. Canonicalize execution_context using RFC 8785 (JCS).

3. Compute hash using declared instruction_hash_alg.

4. Compare computed hash with instruction_hash.

5. Compare computed context hash with execution_context_hash.


If any comparison fails:

• Exit State: HALT

• Reason Code: INSTRUCTION_HASH_MISMATCH or CONTEXT_HASH_MISMATCH

• Side Effect: NOT CALLED

• Nonce: NOT CONSUMED


Terminal.


⸻


Step 6 — Scope & Binding Validation


CEG SHALL verify:

• authority_scope_id covers target_id

• Delegation chain authorizes actor_id

• action_type matches authorized scope


If validation fails:

• Exit State: HALT

• Reason Code: SCOPE_INCONSISTENCY

• Side Effect: NOT CALLED

• Nonce: NOT CONSUMED


Terminal.


⸻


Step 7 — State-Lock Acquisition


CEG SHALL attempt to acquire an Exclusive lock on state_lock_key.


Lock acquisition rules:

• If lock exists AND not expired:

• Exit State: HOLD

• Reason Code: STATE_LOCK_BUSY

• Side Effect: NOT CALLED

• Nonce: NOT CONSUMED

• Lock: NOT MUTATED

• Terminal.

• If lock TTL expired:

• Expire lock.

• Continue acquisition attempt.


If lock successfully acquired:

• Continue to Commit Point.


⸻


3. Commit Point (Atomic Boundary)


The Commit Point SHALL occur only after:

• Steps 1–7 complete successfully.

• Lock acquisition succeeds.


At Commit Point:

• grant_id SHALL be inserted into nonce_registry.

• Lock ownership SHALL be confirmed.

• Both operations SHALL be atomic.


After Commit Point:


The grant is irrevocably consumed.


⸻


4. Execution


Upon successful Commit:

• CEG SHALL execute the side effect.

• No additional validation is performed.

• Lock SHALL be released after execution completes.


If execution fails internally:

• System SHALL log failure.

• Nonce remains consumed.

• Lock SHALL be released.


⸻


5. Deterministic Exit States


Exit State Trigger Class Side Effect Nonce Lock

DENY Schema / Freshness / Replay Not executed Not consumed Not acquired

HOLD Environmental (Lock Busy) Not executed Not consumed Not mutated

HALT Integrity / Scope / Signature Not executed Not consumed Not mutated

EXECUTED All steps passed Executed Consumed Acquired then released



⸻


6. Prohibited Behaviors


CEG SHALL NOT:

• Execute on partial validation.

• Consume nonce prior to Commit Point.

• Infer intent.

• Modify payload.

• Override scope.

• Execute without lock.

• Execute without signature validation.


⸻


7. Falsifiability Condition


The safety model is invalidated if any adversarial vector results in:

• side_effect_called == true

• OR nonce_consumed == true

• prior to full validation and lock acquisition.


⸻


8. Determinism Requirement


Given identical inputs:

• Output state MUST be identical.

• Hash comparison MUST be stable across languages via RFC 8785 canonicalization.

• No nondeterministic branching is permitted.
