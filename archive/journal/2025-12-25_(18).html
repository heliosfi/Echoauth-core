PAGE 18 — TRADE CARD → EXECUTION FLOW (CANONICAL v1.0)


This page defines the exact lifecycle of a trade inside SniperBot:

how a setup becomes a Trade Card, how permission is granted, how execution occurs (manual or auto), and how exits + logging are enforced.


No Trade Card → No Trade

No Stop / TP → No Entry

No Log → No System


⸻


1) Trade Card (The Only Unit of Trade)


A Trade Card is a structured, immutable object created only when whitelist and gate conditions are met.


Trade Card fields (minimum)

• ticker

• timestamp_created

• regime (Trend / Expansion only)

• volatility_bucket (~3.5% / ≥6%)

• news_status (Green / Yellow)

• entry_type (whitelisted)

• entry_trigger (exact condition)

• entry_price_hint (optional)

• stop_level (thesis invalidation)

• tp1_level

• tp2_level

• runner_rules

• risk_percent

• position_size (computed)

• time_valid_until (card expiration)

• auto_eligible (true / false)

• rejection_reasons (if any)


If any required field is missing →

❌ Card is invalid

❌ Card cannot be armed

❌ Card cannot be executed


⸻


2) Trade Card Creation (Setup → Card)


Step A — Detect Setup


SniperBot scans only for:

• Break & retest

• VWAP flip + hold

• Key level reclaim


No other setups are considered.


⸻


Step B — Confirm Context Gates


Must PASS:

• RegimeGate

• NewsGate

• VolatilityGate


If any fail → no card is created.


⸻


Step C — Define Structure

• Stop is placed where the thesis is invalid

• Targets are placed at logical structure zones

• No discretionary placement


⸻


Step D — Size the Trade (RiskGate)


position_size =

(account_equity × risk_percent) / stop_distance


Reject card if:

• Stop distance exceeds allowed max

• Position size too small to execute cleanly

• Liquidity insufficient for size


⸻


Step E — Card Expiration

• Every card has a time validity window

• If trigger not hit before expiration:

• Card auto-cancels

• No execution allowed


⸻


3) Arming Logic (Card → Permission)


Manual Arming

• Bot may alert: “Trigger approaching”

• Human executes manually (inside or outside bot)


Manual mode permissions

• Bot can alert

• Bot can log

• Bot can manage position after it exists (optional)


⸻


Auto Arming (Permissioned Execution)


Auto execution is allowed only if:

• AutoModeGate PASS at arming time

• AutoModeGate PASS again at trigger time


Auto mode permissions

• Execute exactly what is on the card

• Max trades/day enforced (default = 1)

• No parameter changes

• No size changes

• No entry improvisation


If auto eligibility fails at trigger →

Auto-mode is disabled and card remains manual-only.


⸻


4) Execution Flow (Triggered → Order)


Entry Trigger Must Be Exact


Examples:

• “Price closes above level X and holds for N seconds”

• “Retest touches level X and rejects with confirmation”

• “VWAP flip holds + volume threshold met”


If trigger condition is not met exactly → no entry.


⸻


Bracket Order Requirement (Mandatory)


At execution, SniperBot must place:

• Entry order

• Stop-loss order

• TP1 / TP2 orders (or equivalent bracket)


If full bracket cannot be placed immediately:

• ❌ Entry is canceled

• ❌ Trade is rejected


⸻


5) In-Trade Management (After Entry)


Once IN_TRADE:

• Stop cannot be widened

• TP ladder executes automatically

• Runner managed per predefined rules

• Optional time-stop enforced


No new trades may be armed while IN_TRADE (unless explicitly configured).


⸻


6) Exit Conditions


A trade exits only by:

• Stop-loss hit

• TP ladder completion

• Time stop

• Session end

• Governance-triggered shutdown


Forced exits are logged with reason.


⸻


7) Logging & Audit (Non-Negotiable)


Every trade must log:

• Trade Card (full snapshot)

• Execution timestamps

• Fill prices

• Exit reason

• P&L

• State transitions


If logging fails at any point:

• Immediate LOCKOUT

• No further trades allowed


⸻


8) Page 18 Lock


SniperBot does not “take trades.”

It executes approved Trade Cards under governance.


This closes the execution loop.
