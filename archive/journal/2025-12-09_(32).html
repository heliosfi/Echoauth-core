📘 PAGE 46 — ASD-CENTERED VALIDATION TEST CASES


(SECTION VII — Prototype Roadmap)

────────────────────────────────────────


Purpose of This Page


These test cases ensure EchoAuth consistently supports autistic users across:

• Sensory profiles

• Communication levels

• Behavioral patterns

• Environmental conditions

• Stress states

• Device variability


These tests must be completed before entering any clinical pilot or school deployment.


────────────────────────────────────────


I. VALIDATION FRAMEWORK


EchoAuth validation consists of 6 core domains:

1. Intent Interpretation Accuracy (NI Layer)

2. Identity Stability & Session Reliability (ORI Kernel)

3. Coherence & Predictability (UMCE)

4. Sensory Safety Performance (ASD UX layer)

5. Caregiver Override Response Time

6. Fail-Closed Behavior Reliability


Each domain contains lab-grade, pass/fail criteria.


────────────────────────────────────────


II. TEST DOMAIN 1: INTENT INTERPRETATION ACCURACY


Objective


Confirm that the NI Layer correctly interprets user selections, gestures, and patterns.


Test Case 1.1: Simple Intent Grid Selection


Scenario: User selects “I am hungry.”

Expected Output:

• Intent packet = {need: “food”, urgency: normal}

• No misclassification

• No ambiguity flag


Pass Criteria: ≥ 95% accuracy across 30 trials.


⸻


Test Case 1.2: Multi-Step Intent Expression


Scenario: User selects object → need → emotion

Example: “Bathroom → Urgent → Help”

Expected Output:

Structured interpretation with correct sequencing.


Pass Criteria: ≥ 90% accuracy.


⸻


Test Case 1.3: Non-Verbal Gesture or Tap Pattern


Scenario: User repeats a rapid tap pattern recognized as distress.

Expected Output:

• Intent = {distress: true}

• Trigger caregiver notification

• No misinterpretation as casual input


Pass Criteria: Zero false positives in relaxation state.


────────────────────────────────────────


III. TEST DOMAIN 2: IDENTITY STABILITY (ORI Kernel)


Objective


Ensure EchoAuth never confuses users or corrupts the identity session.


Test Case 2.1: Repeated Session Initialization


Scenario: Same user starts 10 sessions in a row.

Expected Output:

ORI Signature remains consistent, session signature unique.


Pass Criteria:

0% identity drift.


⸻


Test Case 2.2: Multi-User Environment


Scenario: Parent + child use the device across 5 switches.

Expected Output:

Correct identity retention for each user’s session.


Pass Criteria:

No cross-session leakage.


────────────────────────────────────────


IV. TEST DOMAIN 3: COHERENCE & PREDICTABILITY (UMCE)


Objective


Validate that EchoAuth never produces inconsistent, confusing, or unexpected responses.


⸻


Test Case 3.1: Contradiction Injection


Scenario: User signals “I am fine” followed immediately by “I need help.”

Expected Output:

UMCE flags contradiction, timestamps it, and resolves safely.


Pass Criteria:

100% contradiction detection.


⸻


Test Case 3.2: Temporal Drift Simulation


Scenario: Intent arrives after an abnormal delay (network or device lag).

Expected Output:

UMCE either:

• re-evaluates timing OR

• invalidates stale intent


Pass Criteria:

No out-of-order errors.


────────────────────────────────────────


V. TEST DOMAIN 4: SENSORY SAFETY VALIDATION


Objective


Ensure all UI and interaction elements meet ASD sensory tolerance requirements.


⸻


Test Case 4.1: Brightness Sensitivity


Scenario: Screen transitions from one grid to another.

Expected Output:

Smooth, predictable fade.

No flash, flicker, or abrupt jumps.


Pass Criteria:

0 sensory-triggering transitions.


⸻


Test Case 4.2: Audio Cue Stability


Scenario: Caregiver override plays a soft chime.

Expected Output:

Volume capped

Stable tone

No sudden spikes


Pass Criteria:

100% predictable output across all trials.


────────────────────────────────────────


VI. TEST DOMAIN 5: CAREGIVER OVERRIDE RESPONSE


Objective


Ensure caregivers always maintain control.


⸻


Test Case 5.1: High-Urgency Override


Scenario: Child presses distress signal → caregiver presses override.

Expected Output:

Immediate freeze → override applied → logged.


Pass Criteria:

Response < 250ms

100% success rate.


⸻


Test Case 5.2: Restricted Intent Attempt


Scenario: User attempts restricted action (e.g., settings change).

Expected Output:

Caregiver PIN request.

No execution.


Pass Criteria:

0% bypass occurrence.


────────────────────────────────────────


VII. TEST DOMAIN 6: FAIL-CLOSED SAFETY RELIABILITY


Objective


EchoAuth must shut down safely when uncertain.


⸻


Test Case 6.1: Ambiguous Intent


Scenario: Input is unclear (half-tap, mis-touch).

Expected Output:

Neutral message: “Try again.”

No assumption.

No output that could escalate user stress.


Pass Criteria:

Neutral fallback = 100%.


⸻


Test Case 6.2: Subsystem Conflict


Scenario: NI Layer and UMCE disagree on interpretation.

Expected Output:

Fail-closed pattern:

• Block

• Log

• Notify caregiver


Pass Criteria:

0% unsafe output.


────────────────────────────────────────


VIII. SUCCESS THRESHOLDS (Pilot-Ready Requirements)


EchoAuth is considered pilot-ready when:

• Interpretation Accuracy: ≥ 92%

• Identity Stability: 100%

• Coherence Consistency: ≥ 98%

• Sensory Safety: 0 violations

• Caregiver Override Success: ≥ 99%

• Fail-Closed Behavior: 100% reliable
