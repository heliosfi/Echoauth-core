PAGE 16 — SUBSYSTEM S1: ORI IDENTITY KERNEL


1. Purpose


The ORI Kernel serves as the root identity anchor for the user.

It provides continuity across sessions without requiring biometrics, making EchoAuth both privacy-preserving and ASD-safe.


2. Core Responsibilities

• Maintain a non-biometric identity signature

• Track user preferences, sensory tolerances, and communication tendencies

• Provide a stable “personality anchor” for the NI-AI interpretation engine

• Ensure predictability across sessions

• Prevent identity drift


3. Inputs


Input Type Description

Session ID Timestamp + device nonce

Preference Vector Known user patterns (colors, layouts, pacing)

Communication Baseline Repeated lexical/gesture patterns

Sensory Profile Brightness, contrast, sound tolerances


4. Outputs


Output Type Description

ORI Signature Deterministic session-personalization key

Preference Hints Informs UI rendering + pacing

Identity Confidence Score Used by NI Layer for stability

Context Frame Carries identity into each subsystem


5. Failure Modes & Handling


Failure Handling

Missing signature Rebuild from defaults; notify caregiver

Corrupted data Reinitialize with last-known-stable snapshot

Low identity confidence Restrict system to basic communication grids


6. Compliance Requirements

• No biometric identifiers stored

• All identity vectors encrypted (AES-256 recommended)

• All updates logged for traceability


7. Engineering Requirements

• Must produce a consistent signature given the same input

• Must not depend on cloud services

• Must expose a simple API: getORI(), updateORI(), validateORI()


⸻


PAGE 17 — SUBSYSTEM S2: INTENT GRID ENGINE (IGE)


1. Purpose


Converts structured inputs—buttons, grids, objects, typed fragments—into deterministic intent candidates.


2. Responsibilities

• Load ASD-specific communication grids

• Provide selectable categories (needs, feelings, routines, safety)

• Resolve ambiguous selections using ORI preferences

• Produce low-variance intent candidates


3. Inputs

• Grid selection index

• Touch event or object selection

• ORI preference vector

• Interaction history (short-term)


4. Outputs


IntentCandidate {

    class: “need_expression” | “emotion” | “routine_request” | “safety_flag”

    detail: string

    confidence: float

    source_grid: string

}


5. Failure Modes


Failure Handling

Multiple conflicting selections Use ORI weighting or prompt user

No selection Timeout → reset grid

Rapid repeated taps Interpret as urgency (NI Layer rule)


6. Compliance

• No free text interpretation

• All grids pre-approved and non-diagnostic

• All transitions predictable


7. Engineering Requirements

• Must function offline

• Must keep latency under 50ms

• Must allow clinics/schools to upload custom grids


⸻


PAGE 18 — SUBSYSTEM S3: NI INTENT CLASSIFIER (NIC)


1. Purpose


Transform IntentCandidates into final structured intents, following ASD-aligned communication templates.


2. Responsibilities

• Map user signals to safe, predictable intents

• Avoid conversational inference

• Enforce clinical communication templates

• Annotate ambiguity levels


3. Inputs

• ORI signature

• IntentCandidate(s)

• Communication Baseline

• Context Frame


4. Outputs


ParsedIntent {

    intent_type: “need” | “emotion” | “request” | “alert”

    intent_value: string

    confidence: float

    ambiguity_flag: boolean

}


5. Failure Modes


Failure Handling

Ambiguous pattern Set ambiguity_flag + route to caregiver

Contradiction Defer to UMCE

Overlapping templates Select highest clinical match


6. Compliance

• No LLM inference

• No self-generating language

• Clinical templates only


7. Engineering Requirements

• Must use rule-based + vector scoring

• No autonomous expansion of intent list

• Deterministic results for identical input


⸻


PAGE 19 — SUBSYSTEM S4: UMCE COHERENCE ENGINE


1. Purpose


Maintain logic, stability, and temporal coherence across system behavior.


2. Responsibilities

• Enforce continuity between previous + current intents

• Detect contradictions

• Apply time-based rules

• Gate outputs that violate coherence or safety


3. Inputs


• ParsedIntent

• ORI signature

• Session Memory (short-window)

• Safety Rules


4. Outputs


CoherentIntent {

    status: “approved” | “blocked” | “needs_caregiver”

    resolved_intent: ParsedIntent | null

    reason: string

}


5. Failure Modes


Failure Handling

Drift Reset to ORI baseline

Contradiction Block + notify caregiver

Impossible request Auto-block


6. Compliance

• Required in all clinical environments

• Must log all blocked intents

• Must be auditable


7. Engineering Requirements

• Deterministic finite-state logic

• Maintain sliding 2–5 min temporal window

• ~0ms inference drift tolerance


⸻


PAGE 20 — SUBSYSTEM S5: EXECUTION SAFETY LAYER (ESL)


1. Purpose


Final authority that determines whether an intent may proceed.


2. Responsibilities

• Apply safety rules

• Apply caregiver permissions

• Enforce fail-closed behavior

• Produce final approved action


3. Inputs

• CoherentIntent

• Safety Policy

• Caregiver Overrides

• Environment Flags


4. Outputs


FinalAction {

    action_type: “communicate” | “notify” | “log”

    payload: string

    caregiver_required: boolean

}


5. Failure Modes


Failure Handling

Safety violation Block + notify

Restricted domain Prompt caregiver

Uncertain confidence Fail-closed


6. Compliance

• Must comply with FERPA/HIPAA

• Must never produce an action outside allowed domains


7. Engineering Requirements

• All rules encoded in static policy files

• No dynamic expansion

• Zero exceptions to safety gating
