9.1 Safety Envelope Architecture (SEA)


The SEA surrounds the entire NI-AI pipeline and acts as the ultimate authority over what the system may or may not do.


SEA enforces:

• Bounded outputs only

• No free-form generation

• No emotional inference outside defined ASD-safe ranges

• Strict identity matching against ORI kernel signatures

• Prevention of unsafe, misleading, or ambiguous communication

• Absolute refusal when rules are violated


Three layers of SEA:

1. Pre-Processing Safety Gate

• Validates identity vector

• Confirms child’s safety level

• Rejects malformed or ambiguous inputs

2. Mid-Pipeline Safety Guardrails

• Blocks unsafe intent interpretations

• Enforces IEP-aligned rules

• Applies cooldown logic (e.g., repeated urgent requests)

3. Post-Processing Safety Filter

• Ensures output is child-safe

• Ensures no hallucinations

• Ensures no unauthorized advice or responses


If ANY layer raises a violation → immediate fail-closed behavior.


⸻


9.2 Fail-Closed Execution (No Exceptions)


Fail-closed =

If EchoAuth is unsure, it refuses.

If EchoAuth cannot validate, it refuses.

If EchoAuth detects risk, it refuses.


This is safer than generative AI, because EchoAuth:

• never “guesses”

• never invents responses

• never delivers unsupported emotional conclusions

• never outputs beyond its safety contract


Fail-Closed Standard Block:


{

  "status": "refused",

  "reason": "safety_violation",

  "action": "blocked",

  "requires_guardian": true,

  "explainability": "Input violated rule S-42: ambiguous intent > 0.4"

}


This is EXACTLY what SBIR reviewers, clinicians, and IRB boards want to see.

9.3 Regulatory Alignment (FERPA • HIPAA • COPPA)


EchoAuth’s architecture is built for compliance by design, not by patching.


FERPA (Schools)


EchoAuth NEVER stores:

• academic history

• grades

• personally identifying education records


EchoAuth ONLY stores:

• structured intent logs

• guardian-approved configurations

• device-level activity


These records remain under institution control, not app control.


⸻


HIPAA (Clinics/Health Settings)


EchoAuth avoids PHI by:

• storing no medical history

• storing no diagnoses

• storing no clinician notes


If integrated into a clinical setting:

• HIPAA-compliant encryption defaults

• audit trails enabled

• patient identifiers hashed via ORI identity vector


⸻


COPPA (Children Under 13)


EchoAuth:

• stores no personal details

• uses no biometric identification

• provides guardian-controlled permissions panel

• allows total data wipe at any time


EchoAuth is inherently COPPA-safe because it does NOT track, mine, or profile children.


⸻


9.4 Guardian Override Logic


A parent, guardian, teacher, or clinician can ALWAYS override the system.


Override permissions include:

• Approve or deny action routes

• Modify escalation recipients

• Adjust sensory UI levels

• Halt all communication output

• Trigger emergency contact notifications


Override actions are logged with:

• timestamp

• device ID

• guardian identity signature


EchoAuth must make all override logs immutable.


⸻


9.5 System-Level Refusal Behavior


The following ALWAYS triggers immediate refusal:


Hard Refusals:

• Identity mismatch (ORI fails to bind user)

• High-risk intent (e.g., danger-to-self or others without guardian authority)

• Ambiguous communication above threshold

• Device integrity failure

• Environmental rule conflict (e.g., school lockdown mode)

• Unauthorized cross-context communication (e.g., school → home)


Soft Refusals:

• Cooldown rule violation

• Repetitive signals inconsistent with user profile

• Attempts to perform actions outside IEP guidelines


Both hard and soft refusals route into the SEA Log + Guardian Notification Pathway.


⸻


9.6 Auditing & Explainability Requirements


EchoAuth must log:

• All validated intents

• All refusals

• All safety rule triggers

• All identity mismatches

• All guardian overrides

• All device health checks


Logs MUST be:

• timestamped

• signed

• immutable

• exportable for IRB / SBIR evaluation

• explainable in plain language for clinicians


Example explainability message:


“Intent rejected because the system detected conflicting gestures (Rule S-17).

Guardian review recommended.”


This is EXACTLY the level of auditability federal programs require.
