⭐ PAGE 10 — PILOT ENGINEERING REQUIREMENTS (Schools, Clinics, Home Programs)


This section defines the exact technical, operational, and environmental requirements needed to safely deploy EchoAuth in a controlled pilot setting.

It ensures predictable behavior, safe data handling, and repeatable testing conditions.


Every pilot site (school, ASD clinic, or therapy program) must agree to the specifications below.


⸻


10.1 Pilot Environment Requirements


Each pilot environment must supply:


1. Stable, Controlled Device Space

• Dedicated tablet or phone (Android/iOS supported)

• Minimum specs:

• 3GB RAM

• 32GB storage

• iOS 15+ or Android 11+

• Disabled background apps

• Device must be used ONLY for EchoAuth during session time.


2. Predictable Environmental Conditions


To reduce sensory overload:

• Moderate lighting (no flicker, no strobe)

• Low-noise room (≤ 50 dB preferred)

• No competing visual stimuli


3. Fixed User Positioning

• User should be within arms-length reach of device

• Device height consistent with IEP or clinician notes

• No sudden environmental changes during session


These constraints ensure the NI-AI engine receives consistent input patterns.


⸻


10.2 Technical Deployment Requirements


1. EchoAuth Application Layer

• Deployed through secure link or MDM (Mobile Device Management)

• No third-party integrations during Phase I

• EchoAuth runs within a restricted sandbox


2. Connectivity Requirements


EchoAuth supports:

• Offline mode (core functions)

• Secure online mode for sync logs only


When online:

• Encrypted TLS connection to HeliosFi secure endpoint

• Zero third-party routing (no external APIs)


3. Guardian / Teacher Access Panel


Each site must enable:

• One Guardian or Staff Admin account

• One fallback override key

• Daily sync of session logs (auto-upload)


4. Local Data Protections

• All data stored in-device using AES-256 encryption

• No data leaves device without guardian / admin approval

• Local logs purged every 30 days unless requested by IRB or pilot evaluator


⸻


10.3 Pilot User Requirements (ASD Participants)


Pilot participation requires:


1. ASD Profile Overview


Clinician or guardian must provide:

• Communication style (verbal, low-verbal, non-verbal)

• Sensory sensitivities

• Typical triggers

• Current communication tools used (PECS, AAC, gestures)


EchoAuth does not store medical history — only communication-relevant parameters.


2. Consent & Safety Verification

• Guardian consent required

• School or clinic must sign pilot MOU

• IRB approval (if applicable)


3. Interaction Time Requirements


Minimum usage:

• 10–15 min/day, 3–5 days/week

• 4-week minimum pilot duration


This ensures enough data to evaluate consistency and reliability.


⸻


10.4 Pilot Evaluation Metrics (Engineer + Clinical Review)


EchoAuth is evaluated using quantitative and qualitative measures.


1. Communication Accuracy Rate


Measured as:

• % of user actions mapped correctly to intended meaning

Goal: ≥ 30% improvement compared to baseline


2. Identity Confirmation Reliability


EchoAuth must:

• Correctly associate ORI Identity Vector with the same user

• Reject any ambiguous identity session

Goal: ≥ 95% identity stability


3. Sensory Load Index


Clinician evaluation of:

• Stress behaviors

• Overstimulation

• User agitation

Goal: measurable reduction during EchoAuth use


4. Guardian/Teacher Satisfaction


Feedback on:

• Ease of use

• Predictability of system

• Ability to understand user needs faster

Goal: 80% positive satisfaction


⸻


10.5 Engineering Validation Requirements


Before a pilot can begin, engineers must confirm:


1. System Health Checks

• ORI Identity Kernel calibration complete

• Coherence engine active

• Safety envelope (SEA) enabled

• No drift tolerance (strict mode)


2. Logging Pipeline Functional

• Logs timestamped and signed

• All refusal events recorded

• All safety triggers recorded

• Guardian overrides recorded

• Export function working (PDF + JSON)


3. Stability Testing

• No crashes in 3-hour idle mode

• No freeze events during 20 interactions

• Fail-closed triggers activate under test conditions


4. UI Accessibility Validation

• Color contrast within ASD-safe norms

• Predictable layout

• Button spacing meets motor-control standards


⸻


10.6 Pilot Staff Training Requirements


Teachers, clinicians, or guardians must complete a 30-minute training module:


Training includes:

1. How to interpret EchoAuth outputs

2. How to modify UI levels for sensory safety

3. How to view logs

4. How override functions work

5. What the system will NOT do (important for safety)


This ensures NO misuse.


⸻


10.7 Prohibited Pilot Configurations


To protect participant safety:


❌ No biometric scanning

❌ No free-form text generation

❌ No camera-based emotion recognition

❌ No integration with student academic systems

❌ No unsupervised usage with minors

❌ No exporting data to consumer apps

❌ No modifications to ORI Identity Vector


These are non-negotiable constraints for federal and clinical compliance.
