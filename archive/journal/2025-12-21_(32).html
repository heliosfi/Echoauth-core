Appendix C — Roles & Feature Alignment


(Child-First, Non-Interpretive, No Drift)


⸻


ROLE 1 — Child (Access User)


Purpose: Regulation, engagement, expression

Authority: None (by design)


Features Available

• Flame Login (state-based entry)

• Sensory programs (visual, sound, rhythm)

• Music modules (predictable, calming)

• Approved programs/games

• Calm / pause modes

• Simple “request help” signals


Features Explicitly NOT Available

• No settings

• No alerts

• No prompts about medication

• No scores, levels, or labels

• No feedback implying success/failure


Design rule:


The child experiences support, never management.


⸻


ROLE 2 — Parent / Guardian (Authority Holder)


Purpose: Decision-making, consent, protection

Authority: Absolute


Features Available

• Child profile creation & management

• Permission control (who can see / do what)

• Approval / decline of recommendations

• Activation of care tasks

• Sensory & music program enable/disable

• Medication schedule entry

• Alert routing control

• Full development timeline view

• Audit log access


Features Explicitly NOT Delegated

• Authority is never auto-shared

• No professional can override parent decisions


Design rule:


Nothing becomes active for the child without a parent action.


⸻


ROLE 3 — Caregiver Delegate (Hands-On Support)


Purpose: Daily care execution

Authority: Task-scoped only


Features Available

• View active tasks

• Receive alerts/reminders

• Log task completion

• Log medication administration (if authorized)

• Add factual notes (“given”, “completed”, “missed”)


Features Explicitly NOT Available

• No approvals

• No permission changes

• No recommendation activation

• No sensory program control


Design rule:


Caregivers act, parents decide.


⸻


ROLE 4 — School Staff (Educational Contributor)


Purpose: Observation, support, documentation

Authority: Advisory only


Features Available

• Log classroom observations

• Upload school documents (IEP, reports)

• Propose supports/accommodations

• Execute approved tasks

• Log outcomes

• View shared development timeline (scoped)


Features Explicitly NOT Available

• No permission changes

• No activation of recommendations

• No medication scheduling

• No sensory module control


Design rule:


Schools contribute insight, not authority.


⸻


ROLE 5 — Clinical Staff (Care Coordination Contributor)


Purpose: Records, recommendations, continuity

Authority: Advisory only


Features Available

• Upload clinical documents (reference only)

• Submit recommendations

• View medication adherence logs (if approved)

• Add clinical notes (non-executing)

• Review development timeline


Features Explicitly NOT Available

• No prescribing logic

• No schedule changes

• No auto-activation of advice

• No alerts sent to child


Design rule:


Clinical input informs care — it does not control it.


⸻


ROLE 6 — System (EchoAuth)


Purpose: Safety, coordination, evidence

Authority: None over humans


Features Executed Automatically

• Confidence gating & pause-on-uncertainty

• Time-based reminders

• Event logging

• Access auditing

• De-identified reporting


Explicit System Limits

• No diagnosis

• No interpretation

• No decision-making

• No optimization of treatment


Design rule:


The system records, reminds, and pauses — it never decides.
