# EchoAuth™ MVP – Caregiver-Governed Runtime (NI-AI Spine)


This repository contains a minimal, attorney-friendly MVP implementation of the EchoAuth™ runtime,

focused on **caregiver-governed decision flow** for non-verbal autistic children.


It does **not** include any production AI model weights.  

Instead, it encodes the **control architecture** and **safety spine**:


- ORI – Identity Anchor Layer (caregiver + child)

- LRM – Load Regulation Matrix (cues → stress score 0–5)

- HER – Runtime Integrity Kernel (NORMAL / SAFE / DEGRADED)

- PNRE – Singular Output Choke-Point (one output label only)

- DAHE – Data Aggregation & History Engine (append-only session history)



## Architecture Spine


The runtime executes in a non-negotiable order:


1. **ORI** – binds `caregiver_id` and `child_id` into an identity context.

2. **LRM** – converts raw non-verbal cues (movement, sound, breathing, etc.) into a discrete stress score in the range 0–5.

3. **HER** – maps the stress score into a runtime mode:

   - `NORMAL`   → low stress

   - `SAFE`     → medium stress

   - `DEGRADED` → high stress

4. **PNRE** – takes internal candidate labels and returns exactly one final label.

5. **DAHE** – logs each check-in and, on high stress, freezes a capsule for later review.


The **Thread Linking Engine (TLE)** is not exposed directly here; it is supported later by DAHE’s history for extended-context recall.



## Safety Principles


- Only **one label** ever reaches the caregiver-facing API response (e.g. `"Safe"` or `"Needs Attention"`).

- Internal reasoning, multiple candidate outputs, and conflict resolution happen **inside PNRE** and are **never exposed** to the child interface.

- History is **append-only** and is always tied to a specific caregiver and child identity.

- In `DEGRADED` mode, the system prioritizes **survival and clarity**: short, stable responses plus an immutable capsule for later human review.



## Files


- `app.py` – FastAPI app with the `/checkins` endpoint (main entry point).

- `models.py` – Shared Pydantic models, HER modes, and data structures.

- `ori.py` – ORI identity anchor logic for caregiver + child.

- `lrm.py` – Simple Load Regulation Matrix that scores stress 0–5 from cues.

- `her_kernel.py` – HER mode selection + basic config hints.

- `pnre.py` – PNRE singular-output gate (chooses one label).

- `dahe.py` – In-memory DAHE implementation (event log + capsule storage).



## API Overview


### POST `/checkins`


**Request body:**


```json

{

  "caregiver_id": "caregiver-123",

  "child_id": "child-456",

  "cues": {

    "movement": "pacing",

    "sound": "humming",

    "breath": "fast"

  }

}


Response body (example):


{

  "label": "Needs Attention",

  "her_mode": "SAFE",

  "stress_score": 3,

  "capsule_id": null,

  "timestamp": "2025-11-29T12:34:56.789Z"

}


• label – final caregiver-facing decision (no contradictions, no internal threads).

• her_mode – runtime mode chosen by HER (NORMAL, SAFE, or DEGRADED).

• stress_score – integer 0–5 from LRM.

• capsule_id – present when a high-stress (DEGRADED) event is frozen for audit.

• timestamp – UTC time of the result.


Running the MVP

1. Install dependencies:


pip install fastapi uvicorn pydantic


2. Run the API:


uvicorn app:app --reload


3. Send a test request (via curl, Postman, or any HTTP client).


This MVP is intentionally simple and transparent so that attorneys, reviewers, and engineers

can trace every decision from cue input to final label with no hidden behavior.

EchoAuth™ MVP Pilot Checklist


Goal:

Deploy a minimal, caregiver-governed runtime that turns real-world cues from non-verbal autistic children

into singular, safe, auditable outputs, aligned with the EchoAuth patent architecture.


1. Identity & Access

   - [ ] Caregiver account creation and authentication are in place.

   - [ ] Each child has a unique EchoAuth child_id bound to a caregiver_id.

   - [ ] ORI identity context (caregiver_id + child_id) is passed into every /checkins call.


2. Cue Collection

   - [ ] Caregiver-facing UI (or form) supports input of:

       - movement (e.g., pacing, still, flapping)

       - sound (e.g., quiet, humming, crying)

       - breathing (e.g., steady, fast)

       - optional notes (free text for later clinical review)

   - [ ] Cues are sent as a JSON object to the backend without images or videos at this stage.


3. LRM – Stress Scoring

   - [ ] LRM is called on every check-in to compute a stress score (0–5).

   - [ ] Heuristics are simple, explainable, and documented.

   - [ ] Stress scores are logged as part of the DAHE event payload.


4. HER – Mode Selection

   - [ ] HER maps stress score to:

       - NORMAL (0–1)

       - SAFE (2–3)

       - DEGRADED (4–5)

   - [ ] Mode is included in the /checkins result.

   - [ ] HER mode determines any future model config (e.g., max_tokens, temperature) if AI is added.


5. PNRE – Single Output

   - [ ] Internal reasoning (even if simulated) may produce multiple candidate labels.

   - [ ] PNRE always collapses candidates to exactly one final label.

   - [ ] No conflicting or multi-thread outputs are ever shown to caregiver or child.


6. DAHE – History & Capsules

   - [ ] All check-ins create an EventRecord containing:

       - identity (caregiver_id, child_id)

       - cues

       - stress_score

       - her_mode

       - final label

       - timestamp

   - [ ] High-stress (DEGRADED) events generate a capsule_id.

   - [ ] Capsules are append-only and retrievable by authorized caregiver/clinician.


7. UI & Safety

   - [ ] Child-facing surfaces never show internal reasoning, logs, or multiple labels.

   - [ ] Caregiver sees only:

       - final label

       - optional brief explanation (if added later)

   - [ ] No "dark boxes", diagrams, or visual artifacts from development appear in caregiver UI.


8. Legal & Documentation

   - [ ] Patent specification skeleton (EchoAuth_Patent_Pack_Final.pdf) is attached to the project folder.

   - [ ] README clearly explains:

       - ORI → LRM → HER → PNRE → DAHE spine

       - safety principles

       - embodiment nature (non-claim-altering)

   - [ ] Claims 1–420 remain stored separately and unchanged.


9. Pilot Review Loop

   - [ ] Regular review sessions scheduled with caregivers and clinicians.

   - [ ] Feedback loop on:

       - usefulness of labels

       - clarity of UI

       - reliability of stress scoring

   - [ ] Any changes to LRM/HER/PNRE/DAHE are logged and versioned for legal traceability.
