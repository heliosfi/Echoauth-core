Technical Appendix


Structural Mechanisms for Human-Safe Authorization


(Supplement to Human-Safe Authorization: Preventing AI Action Under Ambiguity)


⸻


A. System Framing: Authorization as a Control Problem


Human-Safe Authorization treats AI-assisted decision systems as control systems, not prediction engines.


In this framing:

• AI models generate signals, not actions

• Authorization is a state-transition problem

• Safety is enforced through invariants, not accuracy thresholds


The goal is not to improve prediction under uncertainty, but to prevent irreversible transitions while uncertainty remains.


⸻


B. Control Loop Architecture


The system is modeled as a bounded control loop with explicit states:


[ STABLE CONTEXT ]

        ↓

[ RECOGNITION ]

        ↓

[ ATTENTION GATED ]

        ↓

[ INTENT CONFIRMED ]

        ↓

[ AUTHORIZATION CHECK ]

        ↓

[ ACTION | DENY | PAUSE ]


Key Properties

• Transitions are unidirectional

• No state skipping is permitted

• Each transition requires explicit satisfaction of constraints


This eliminates silent escalation paths.


⸻


C. Pause-on-Uncertainty as a First-Class State


Unlike traditional systems where pause is an exception or timeout, Human-Safe Authorization defines PAUSE as a valid, persistent state.


Conditions that trigger PAUSE

• Conflicting policy rules

• Missing authority confirmation

• Ambiguous intent signals

• Context mismatch

• Sensor or input uncertainty


Properties of PAUSE

• No automatic resolution

• No background inference

• No decay into action

• Requires explicit upstream clarification


Pause is treated as containment, not failure.


⸻


D. Authority Separation (Structural, Not Procedural)


Authority is enforced in code, not through policy documents or human memory.


Role Separation Example

• Signal Producer (AI / sensors): may observe, infer, suggest

• Intent Holder (human operator / parent / authorized role): confirms

• Executor (system): acts only after authorization


Enforcement Mechanisms

• Role-bound APIs

• No shared credentials

• No cross-role method calls

• No override paths without re-authorization


This prevents responsibility collapse.


⸻


E. Deterministic State Transitions


Human-Safe Authorization prohibits probabilistic transitions at the authorization boundary.


Rules

• AI outputs cannot directly trigger action

• Probabilities are inputs, not decisions

• Authorization outcomes are binary:

• GRANT

• DENY

• PAUSE


This ensures predictability and auditability.


⸻


F. Deny-by-Default Invariant


If required conditions are not explicitly satisfied, the system defaults to DENY or PAUSE, never to action.


This invariant applies to:

• incomplete policy definitions

• missing authority confirmation

• degraded system state

• ambiguous context


This contrasts with many AI systems that default to “best guess.”


⸻


G. Audit-First Design


Every attempted transition generates an immutable audit event before action is taken.


Minimum Audit Fields

• timestamp

• requesting role

• target resource

• current system state

• policy version

• outcome (grant / deny / pause)

• reason code


Audit logs are:

• append-only

• non-mutable

• role-restricted for viewing


This preserves accountability under stress and post-incident review.


⸻


H. Reduction of Exploit Surfaces


Human-Safe Authorization reduces exploitability by collapsing common attack vectors:


Typical Exploit Vector Structural Mitigation

Prompt injection No direct action from model output

Over-confidence bias Explicit confirmation required

Authority spoofing Role-bound execution paths

Ambiguity exploitation Pause-on-uncertainty

Silent escalation Deterministic state machine


The system does not attempt to detect malicious intent; it removes the pathways malicious intent relies on.


⸻


I. Relationship to Existing AI Safety Techniques


Human-Safe Authorization is complementary to:

• model alignment

• red-teaming

• interpretability

• monitoring


It operates upstream of all of them, by constraining when model outputs can influence the world.


⸻


J. Key Technical Insight


Safety does not emerge from better prediction alone.

It emerges from preventing action when prediction is insufficient.


This appendix demonstrates how that principle can be implemented structurally, independent of model architecture or domain.
