PAGE 21 — SUBSYSTEM S6: TEMPORAL CONTEXT ENGINE (TCE)


1. Purpose


Maintain time-based understanding of user interactions, emotional pacing, and intent windows.


2. Responsibilities

• Track short-range temporal patterns (few seconds to minutes)

• Stabilize interpretation during variable pacing (common in ASD)

• Reset context after inactivity

• Prevent cross-intent contamination


3. Inputs

• Timestamped events

• ParsedIntent stream

• ORI session signature

• Interaction cadence pattern


4. Outputs


TemporalFrame {

    window_id: string

    recency_score: float

    reset_flag: boolean

}


5. Failure Modes


Failure Handling

Long inactivity Reset window, preserve ORI

Rapid bursts Flag urgency for NI Layer

Overlapping windows Select most recent origin point


6. Compliance

• No predictive inference

• No extrapolated emotion

• All timing rules deterministic


7. Engineering Requirements

• Sliding window 5–180 sec

• State resets must be logged

PAGE 22 — SUBSYSTEM S7: CONTEXT WINDOW MANAGER (CWM)


1. Purpose


Manage what the system “pays attention to” in each moment.


2. Responsibilities

• Activate/deactivate context frames

• Limit noise from unrelated inputs

• Maintain a stable interaction scope

• Prevent cognitive overload


3. Inputs

• TemporalFrame

• ORI baseline

• Active grid

• Interaction history


4. Outputs


ContextScope {

    active_domain: “needs|emotion|routine|safety”

    expected_responses: list

    drop_noise: boolean

}


5. Failure Modes


Failure Handling

Too many context signals Narrow to primary domain

Contradictory cues Defer to UMCE

Unexpected modality Reset scope


6. Compliance

• Required for ASD-friendly cognitive stability

• Must be deterministic


7. Engineering Requirements

• Latency < 5ms

• Context domains must be predefined

PAGE 23 — SUBSYSTEM S8: SEMANTIC DEPTH ENGINE (SDE)


1. Purpose


Provide depth scoring without inference beyond ASD-approved patterns.


2. Responsibilities

• Evaluate intent nuance

• Score clarity vs ambiguity

• Provide “depth indicators” without AI speculation


3. Inputs

• ParsedIntent

• ContextScope

• ORI signature


4. Outputs


DepthProfile {

    clarity: float

    ambiguity: float

    template_match: float

}


5. Failure Modes


Failure Handling

Low clarity Route to caregiver

High ambiguity Trigger re-selection prompt

No template Default to low-depth profile


6. Compliance

• Cannot interpret tone, sarcasm, or complex emotion

• All templates human-designed


7. Engineering Requirements

• Must run offline

• No machine learning models permitted

PAGE 24 — SUBSYSTEM S9: EMOTIONAL STATE CLASSIFIER (ESC)


1. Purpose


Provide non-diagnostic emotional state categories based solely on structured patterns.


2. Responsibilities

• Detect urgency via interaction speed patterns

• Detect frustration via repeated resets

• Detect calm via slow, deliberate interactions


3. Inputs

• Tap cadence

• Grid switching frequency

• Touch pressure (if device supports)

• SDE depth scores


4. Outputs


EmotionalState {

    level: “calm|neutral|frustrated|urgent”

    confidence: float

}


5. Failure Modes


Failure Handling

Ambiguous signals Default to neutral

High-urgency spike Notify caregiver

Erratic input Reduce UI complexity


6. Compliance

• Not a diagnostic tool

• No emotion prediction

• Must follow FERPA/HIPAA guidelines


7. Engineering Requirements

• Only structured signals allowed

• 100% deterministic algorithm

PAGE 25 — SUBSYSTEM S10: TEMPORAL MEMORY ENGINE (TME)


1. Purpose


Store short-term communication patterns to stabilize interactions.


2. Responsibilities

• Maintain last 3–7 intents

• Provide user pacing pattern

• Support UMCE continuity

• Assist in routine recognition


3. Inputs

• CoherentIntent

• ORI signature

• EmotionalState


4. Outputs


TemporalMemorySnapshot {

    last_intents: list

    avg_cadence_ms: float

    routine_hint: optional

}


5. Failure Modes


Failure Handling

Memory overflow Auto-trim list

Contradictory chain Notify UMCE

Corrupted frame Rebuild snapshot


6. Compliance

• All snapshots encrypted

• Not stored long-term unless caregiver approves


7. Engineering Requirements

• FIFO queue

• Max length configurable

PAGE 31 — SUBSYSTEM S16: STATE RECONCILIATION ENGINE (SRE)

Resolve inconsistencies between NI, UMCE, and Safety layers.


2. Responsibilities

• Reconcile mismatched subsystem outputs

• Route unresolved states to caregiver


3. Inputs

• CoherentIntent

• EmotionalState

• StabilityProfile


4. Outputs

    recommended_action: string

}


5. Failure Modes


Failure Handling

Unresolvable conflict Caregiver decision required

System uncertainty Fail-closed


6. Compliance

• Must log every conflict

• No automated inference


7. Engineering Requirements

• Weighted conflict resolution tables
