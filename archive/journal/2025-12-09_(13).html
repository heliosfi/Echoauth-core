PAGE 30 — Integration Overview


EchoAuth integrates into existing educational, clinical, or home systems using a three-layer interaction model:


1. Application Layer (External System)


The host system (school tablet app, clinic dashboard, caregiver app) sends:

• Session start requests

• User inputs (taps, selections, objects)

• Configuration updates

• Caregiver overrides


2. EchoAuth API Layer


Provides the stable interface for all external components:

• /session/start

• /input/submit

• /config/update

• /caregiver/override

• /log/export


3. Internal Subsystem Layer


Handles interpretation, coherence, safety, and output.


Data never bypasses:

ORI → NI → UMCE → ESL → Output


Integration Rules

• No direct subsystem access allowed

• All communication passes through API layer

• All outputs enforced through Safety Wrapper

• All override operations logged


This ensures predictable behavior across all deployments.

PAGE 31 — API Endpoints


1. /session/start — Start New Session


Method: POST

Input:


{

  "device_id": "string",

  "user_id": "optional string"

}


Output:


{

  "session_token": "string",

  "ori_signature": "hash",

  "status": "initialized"

}



⸻


2. /input/submit — Submit User Input


Method: POST

Input:


{

  "session_token": "string",

  "input": {

    "type": "grid | text | object",

    "value": "symbol_or_text"

  }

}


Output:


{

  "intent": "structured_object",

  "confidence": "0–1",

  "coherence_state": "valid | blocked",

  "safety_state": "allowed | blocked"

}



⸻


3. /config/update — Update System Configuration


Method: POST

Input:


{

  "session_token": "string",

  "config": {

    "grid_layout": "...",

    "templates": "...",

    "policies": "..."

  }

}


Output:


{ "status": "updated", "version": "string" }



⸻


4. /caregiver/override — Apply Override


Method: POST

Input:


{

  "session_token": "string",

  "pin": "string",

  "action": "approve | block | modify"

}


Output:


{ "override_status": "applied", "logged": true }



⸻


5. /log/export — Retrieve Logs


Method: GET

Output: Encrypted JSON log bundle


{

  "session_logs": [...],

  "timestamp": "ISO"

}


Minimal surface → maximum safety.

PAGE 33 — Environment Configuration


Device Environment

• Touchscreen required

• Encryption enabled

• Local storage available

• Offline mode supported


Software Environment


Python ≥ 3.10 (or compiled runtime equivalent)

Local config: config.yaml, safety_policy.yaml

SQLite / local store for logs

TLS for any external connection


Runtime Variables

• ECHOAUTH_MODE=local | hybrid | enterprise

• SAFETY_POLICY_PATH

• ORI_STORAGE_PATH

• LOG_EXPORT_DIR


API Security

• Local deployments: PIN-based

• Enterprise deployments: SAML/OIDC


Build Requirements

• No GPU required

• No external model calls

• Everything must run deterministically

PAGE 34 — Test Harness Instructions


The Test Harness validates the full pipeline:


1. Unit Tests

• ORI signature loading

• Intent parsing

• UMCE coherence checks

• Safety rule evaluation


2. Integration Tests


Simulate full input → ORI → NI → UMCE → ESL → Output.


3. ASD-Specific Behavioral Tests

• Same input repeated → same output (consistency test)

• Low-sensory mode test

• Grid mis-tap recovery test

• Timeout / overwhelm scenario


4. Fail-Closed Tests


Confirm that ambiguous or contradictory inputs ALWAYS block.


5. Override Tests

• Valid PIN → approved

• Wrong PIN → logged and rejected


Outcome:

System must pass all tests before pilot deployment.
