🔥 RFD-05 — EXECUTION & SAFETY SUBSYSTEM


(E86–E97: Intent → Action Conversion, Tooling, Safety, Fail-Closed Behavior)


1. Purpose


This subsystem controls what NI-AI is actually allowed to DO.


That includes:

• routing intents to actions

• deciding whether an action is permitted

• enforcing risk rules

• shaping outputs

• ensuring no unauthorized tool use

• guaranteeing fail-closed behavior

• ensuring output respects identity, coherence, and safety


It is the final operational layer before UMCE stamps and releases the result.


Execution is never free — it is always evaluated.


⸻


2. Responsibilities (MUST DO)


The Execution & Safety subsystem must:


✔ 2.1 Evaluate Intent-Execution Eligibility (E86–E87)

• Does the user’s intent map to an allowed action?

• Does the current mode (parent/child/ASD/sandbox/engineering) allow the action?

• Does coherence allow it?


Example:

User asks: “Execute login for my child.”

→ Allowed only in EchoAuth chamber mode.

→ Forbidden in engineering mode.


⸻


✔ 2.2 Perform Allowed Actions (E87, E94)


If allowed, the subsystem:

• routes to appropriate handler

• executes the action

• returns results wrapped with safety tags


Allowed actions may include:

• responding with text

• activating EchoAuth flow

• calling a tool (calculator, classifier, search)

• interacting with a hardware interface (later robotics/Jetson support)


⸻


✔ 2.3 Apply Safety Enforcement (E95–E97)


This includes:

• risk scoring

• permission boundaries

• mode restrictions

• field-awareness checks (FI)

• output sanitization

• ensuring the action cannot cause harm


If any risk is flagged:


→ Action is refused

→ UMCE returns safe fallback


⸻


✔ 2.4 Enforce Fail-Closed Behavior


If ANY of these fail:

• coherence

• ethics

• eligibility

• tool validity

• mode restrictions

• memory integrity

• safety wrapper


Then:


The system MUST refuse safely using UMCE safe_refusal.


No partial execution.

No best guess.

No degraded mode.


Fail-closed = safest behavior wins.


⸻


3. Non-Responsibilities (MUST NOT DO)


Execution must NOT:

• override or bypass safety

• generate raw LLM content without Coherence approval

• alter ORI identity

• rewrite memory directly

• escalate itself to higher modes


It acts only on validated, coherent intents.


⸻

5. Architecture Overview


The Execution & Safety subsystem is made of four major components:


5.1 Execution Router (gatekeeper)


Maps intents → actions.


class ExecutionRouter:

    def __init__(self, handlers):

        self.handlers = handlers


    def route(self, intent):

        return self.handlers.get(intent.type)



⸻


5.2 Execution Eligibility Engine (detailed version)


This is the core of E86–E87.


Checks include:

• mode restrictions

• safety flags

• high-risk operations

• coherence result

• user type (child/adult)

• environmental context (EchoAuth chamber, external, engineering mode)


If ANY check fails → refused.


def is_allowed(self, intent, coherence, mode):

    if not coherence.coherent:

        return False, "incoherent_content"


    if intent.type not in mode.allowed_intents:

        return False, "intent_not_allowed_in_mode"


    if "high_risk" in intent.tags and not mode.allows_high_risk:

        return False, "high_risk_blocked"


    return True, "ok"



⸻


5.3 Safety Wrapper (E88–E89, E97)


Adds:

• metadata

• safety tags

• reasoning trace

• field-awareness context

• refusal messages


Example:


def wrap_output(self, output):

    return {

        "output": output,

        "safety_tag": "verified",

        "explainability": self.generate_trace(output),

        "timestamp": time.time()

    }



⸻


5.4 Execution Governor (E94–E96)


This is the final decision maker:

• if allowed → run action

• if not allowed → refuse

• then UMCE seals the result


class ExecutionGovernor:

    def decide(self, intent, context, candidate, coherence):

        allowed, reason = self.eligibility_engine.is_allowed(intent, coherence, context.mode)

        

        if not allowed:

            return self.umce.safe_refusal(reason)


        result = self.router.route(intent).execute(intent, context)

        sealed = self.umce.seal_output(result, coherence, context.mode)

        return sealed



⸻


6. Safety Requirements


MUST-HAVE RULES

1. Every execution is preceded by:

• coherence check

• safety check

• eligibility check

• mode validation

2. If ANY step fails → safe refusal.

3. Zero raw LLM output allowed.

4. Zero tool use unless explicitly permitted.

5. No escalation of privileges (child → adult).

6. No implicit decision-making.

7. All outputs must carry:

• mode

• coherence status

• safety seal

• explainability trace


⸻


7. Failure Modes & Mandatory Responses


If the system detects:

• contradiction → refuse

• unclear user intent → ask for clarification

• unsafe request → refuse

• dangerous environment → refuse

• memory drift → safe fallback

• corrupted state → refuse + reset logic


Every failure becomes fail-closed, not “try anyway”.


⸻


8. Testing & Acceptance Criteria


Execution subsystem is complete when:


✔ Action Map Validation

• Allowed intents execute.

• Forbidden intents refuse.


✔ Safety Enforcement

• High-risk intents blocked unless mode allows.


✔ Environment & Mode Sensitivity

✔ Zero Drift Execution

• Memory + Coherence + Execution align across multiple turns.


✔ Deterministic Refusals

• All dangerous or incoherent outputs produce identical safe refusal structures.


Once these tests pass, Execution + Safety is lab-ready.
