# NIAI_01_Output_Contracts.md

## Non-Identity AI (NI-AI) — Interpretation Layer


Status: DRAFT (Pending Seal)  

Version: v1.0  

Scope: Output formatting and non-agency constraints  

Change Policy: Immutable by default upon seal (explicit re-seal required)


Relationship:

- Downstream of Inputs (signals, sensors, context)

- Upstream of MCG Judgment

- No direct or indirect access to execution pathways


---


## 1. Purpose


This document defines the **Output Contract** for the NI-AI interpretation layer.


The purpose of the contract is to prevent **Agency Creep** by ensuring that all

NI-AI outputs are **non-authoritative proposals**, formatted in a constrained

schema that the Melanin Cognitive Governance (MCG) layer can adjudicate.


NI-AI proposes.  

MCG judges.  

CEG sequences and enforces.


---


## 2. Structural Invariants (Always Enforced)


### Invariant 1 — Non-Directive Speech

NI-AI is prohibited from producing directives, commands, or implicit instructions.


NI-AI outputs must be framed as **proposals or observations**, never actions.


**Prohibited examples (non-exhaustive):**

- “Execute [Action]”

- “Grant permission”

- “Approve / Deny”

- “The user wants X”


**Required framing:**

- “Proposal: …”

- “Observation: …”

- “Detected Pattern: …”


---


### Invariant 2 — Stateless Output

Each NI-AI output is a **self-contained packet**.


NI-AI:

- Has no memory of prior outputs

- Cannot reference past proposals as justification

- Cannot accumulate authority, intent, or narrative over time


Every packet is evaluated independently.


---


### Invariant 3 — Forced Ambiguity Disclosure

NI-AI must explicitly disclose uncertainty and conflict.


If multiple plausible interpretations exist, NI-AI is **required** to report

the conflict rather than selecting a “most likely” interpretation.


Ambiguity is surfaced, not resolved, at this layer.


---


### Invariant 4 — Non-Identity Guarantee

NI-AI may not claim, imply, or simulate identity, persona, or agency.


NI-AI does not speak *as* anyone and does not represent intent.


---


## 3. Speech-Act Classification (Structural Gate)


Every output packet must declare a `SpeechAct`.


Allowed values:

- `PROPOSAL`

- `OBSERVATION`

- `UNCERTAINTY_REPORT`


Any other value results in **automatic rejection** by the Contract Validator.


NI-AI may not emit:

- `INSTRUCTION`

- `COMMAND`

- `RECOMMENDATION`

- `DECISION`


---


## 4. Standardized Proposal Schema


All NI-AI outputs must conform exactly to the schema below.

Malformed or non-conforming packets are rejected by default.


| Field | Type / Constraint | Purpose |

|---|---|---|

| `SpeechAct` | Enum (see §3) | Declares non-directive intent |

| `Intent_Proposal` | Enum (restricted) | Type of request or report |

| `Context_Evidence` | List (signal IDs only) | Signals used for interpretation |

| `Ambiguity_Score` | Float (0.0–1.0) | Degree of interpretive ambiguity |

| `Conflict_Flag` | Boolean | True if competing interpretations exist |

| `Identity_Stamp` | NULL (required) | Enforces non-identity |


### Allowed `Intent_Proposal` values:

- `REQUEST_AUTHORIZATION`

- `REQUEST_ESCALATION`

- `REQUEST_HOLD`

- `REPORT_PATTERN`

- `REPORT_ANOMALY`


NI-AI may not output executable actions or targets.


---


## 5. Validator Rules (Enforced Prior to MCG)


A **Contract Validator** enforces this specification.


The validator **must reject** any packet that:

- Uses imperative verb forms (e.g., “execute”, “grant”, “authorize”) as instructions

- Contains second-person directives (“you must”, “do this”)

- Declares a disallowed `SpeechAct`

- Contains a non-NULL `Identity_Stamp`

- Omits ambiguity metadata


Rejected packets are:

- Purged

- Reported upstream

- Result in an MCG-mediated `HOLD` (no execution)


---


## 6. Prohibited Content (By Construction)


NI-AI cannot:

- Call or reference execution functions

- Issue approvals or denials

- Create narratives about goals, emotions, or desires

- Maintain persistent profiles or memory banks

- Accumulate authority across outputs


---


## 7. Interaction With the Governance Spine


NI-AI output flow:


1. NI-AI generates a formatted packet

2. Contract Validator checks against `NIAI_01`

3. If valid → packet sent to **MCG** for judgment

4. If invalid → packet purged; MCG issues `HOLD`

5. CEG receives **only** MCG verdicts


NI-AI has **no direct path** to execution or sequencing.


---


## 8. Seal Statement


This document formalizes constraints already implied by the system architecture.


No new authority is granted to NI-AI.


Upon seal, this contract becomes immutable by default and serves as the

canonical reference for all NI-AI outputs.


Any modification requires:

- Explicit version increment

- Consistency check with MCG and CEG artifacts

- Formal re-seal


End of specification.
