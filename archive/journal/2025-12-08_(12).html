⭐ PAGE 36 — PRIVACY & DATA PROTECTION POLICY


(FERPA-Ready • HIPAA-Aligned • COPPA-Compliant • Non-Biometric)


EchoAuth is designed from the ground up as a privacy-first, identity-safe assistive technology for autistic and neurodivergent users.

No personal data is required for the system to function.

No biometric data is ever collected, processed, or stored.


This policy defines how data is treated during system use, testing, deployment, and engineering.


⸻


SECTION A — DATA MINIMIZATION (THE CORE RULE)


EchoAuth follows a strict data minimization philosophy:


✔ The system collects the minimum possible information required for communication support.


✔ No sensitive personal data is required.


✔ EchoAuth does not store conversation logs unless explicitly enabled by guardian consent.


✔ The system never retains raw audio, video, images, or biometrics.


This aligns with:

• FERPA §99.31

• HIPAA Privacy Rule (45 CFR 164.502(b))

• COPPA minimal data principles


⸻


SECTION B — DATA TYPES (WHAT WE DO AND DO NOT COLLECT)


B1. Data EchoAuth Collects


These are non-identifiable, ephemeral, and session-bound:

• Interaction type (button-selection, grid-selection, text input)

• System performance metrics (speed, accuracy, errors)

• Session state (to maintain communication flow)

• Guardian-approved profile settings (icon choices, sensitivity levels)


No user identity is stored inside the system.


⸻


B2. Data EchoAuth Does NOT Collect


EchoAuth will NEVER collect or process:


❌ Biometrics

• facial recognition

• fingerprint scans

• iris scans

• voiceprints


❌ Geolocation

• GPS

• WiFi triangulation

• location tracking


❌ Behavioral surveillance data

• sentiment scores

• emotional state prediction

• behavior risk analysis

• mood classification


❌ Personal identifiers

• full name

• address

• date of birth

• social security number

• phone number


❌ Medical/psychological records


EchoAuth is not a diagnostic system.


⸻


SECTION C — DATA STORAGE RULES


C1. Default Mode: Zero Retention


By default:

• No user data is saved.

• All session data is erased at session end.

• Cache clears immediately.


This makes EchoAuth one of the lowest-liability systems in the ASD assistive tech space.


⸻


C2. Guardian-Enabled Logging (Optional)


Families or clinics may enable limited logging:

• communication attempts

• confirmed selections

• user progress metrics


Logs are always:

• encrypted

• anonymized

• exportable

• deletable at any time


The guardian holds full control.


⸻


C3. Protected Storage (If Enabled)


If logging is activated:

• stored locally or in a FERPA/HIPAA-protected environment

• all storage uses AES-256 encryption

• no cloud transfer without explicit consent


⸻


SECTION D — DATA ACCESS & GOVERNANCE


D1. Role-Based Access Controls (RBAC)


Access levels:

• Guardian Level: full oversight

• School/Clinician Level: limited functional access

• User Level: no access to system admin controls


No third parties may access data.


⸻


D2. No Data Sharing Without Consent


EchoAuth will never:

• sell data

• share data for research

• share data with advertisers

• integrate with external systems


Unless:

• guardian provides signed consent

• data is fully de-identified

• use is IRB-approved (for research only)


⸻


D3. No External System Connections


EchoAuth cannot be linked to:

• school surveillance cameras

• behavior-monitoring tools

• predictive analytics systems

• third-party AI models


This preserves user dignity and safety.


⸻


SECTION E — FERPA, HIPAA, COPPA ALIGNMENT


E1. FERPA


EchoAuth is FERPA-compliant because:

• no educational records are stored by default

• no identifiable student data is processed

• parents control all optional data


E2. HIPAA


EchoAuth is HIPAA-aligned because:

• no Protected Health Information (PHI) is required

• optional clinical logs are encrypted and guardian-controlled


E3. COPPA


EchoAuth is COPPA-compliant because:

• no personal information from minors is collected

• guardian oversight is required

• system functionality does not depend on identity


⸻


SECTION F — USER RIGHTS


Users and guardians may:

• request all stored data

• delete all data

• disable all logging

• export logs securely

• revoke consent at any time


EchoAuth must comply immediately — no delay.
