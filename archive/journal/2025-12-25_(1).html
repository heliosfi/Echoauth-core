EchoAuth passwordless login interface


This is what you can say in funding rooms and also what we can implement later.


What “passwordless” means here


No memorized passwords. Login is a multi-signal confirmation of the person, the device, and the moment.


The allowed login signals (pick 2–4 per user profile)


A) Something you are

• Face ID / fingerprint (device-native biometric)

• Voice phrase (short, user-chosen)

• Optional: “presence/attention” check (very light: blink/pose or mic confirm)


B) Something you have

• Trusted device fingerprint (phone/tablet)

• Secure session token / passkey (platform passkey, if used)

• Proximity confirmation (phone nearby, QR tap, Bluetooth—optional)


C) Something you do

• Typed intent phrase (like we defined): “HYBRID MODE CONFIRMED”

• Pattern gesture (draw shape / tap rhythm)

• Simple challenge-response (2–3 tap choices)


D) Context safety checks

• Time window (session expires, re-auth required)

• Location class (optional: “known environment” vs unknown)

• Risk tier (higher-risk actions require stronger confirmation)


Interface flow (simple)

1. Choose profile (Parent / Caregiver / Operator)

2. Confirm with 2–4 signals (based on profile)

3. Session opens → system starts in PAUSE

4. Access granted to the panel (no execution permission by itself)


Lock sentence: EchoAuth grants access, SniperBot grants permission.

✅ STAGE D — PAPER → HYBRID → AUTO TOGGLE MAP (CANONICAL v1.0)


This is the “don’t accidentally auto-trade” rail.


D1) The Toggles (only these exist)

• SYSTEM_POWER: OFF / ON

• MODE_SELECT: PAPER / HYBRID / AUTO

• ECHOAUTH_SESSION: INACTIVE / ACTIVE

• ARM_MANUAL: OFF / ON

• ARM_AUTO: OFF / ON

• KILL_SWITCH: OFF / ON


D2) Hard Invariants (non-negotiable)

1. ECHOAUTH_SESSION = INACTIVE → STATE = PAUSE

2. KILL_SWITCH = ON → STATE = LOCKOUT

3. MODE_SELECT ≠ AUTO → ARM_AUTO is disabled

4. ARM_AUTO cannot exist unless ARM_MANUAL is available (human control baseline)

5. Auto execution requires:

• MODE_SELECT = AUTO

• ECHOAUTH_SESSION = ACTIVE

• STATE = READY

• Card VALID + auto_eligible

• Auto gates pass at arming and at trigger

6. Logging failure → LOCKOUT (always)


D3) What each mode allows


MODE = PAPER


✅ Create cards

✅ Evaluate grains/gates

✅ Log everything

❌ No execution (manual or auto)

Buttons enabled: DISARM, END SESSION, KILL SWITCH


MODE = HYBRID


✅ Create cards

✅ Alert + log

✅ Human can ARM_MANUAL

❌ ARM_AUTO disabled

Buttons enabled: ARM_MANUAL, DISARM, END SESSION, KILL SWITCH


MODE = AUTO (permissioned, rare)


✅ All HYBRID features

✅ ARM_AUTO available only when conditions met

Auto limits (default):

• 1 trade/day

• risk ≤ 0.5%

• one setup type

• auto disables after 1 trade or any failure


Buttons enabled: ARM_MANUAL, ARM_AUTO, DISARM, END SESSION, KILL SWITCH


D4) Session ignition sequence (every day)

1. EchoAuth login → ECHOAUTH_SESSION = ACTIVE

2. SYSTEM_POWER = ON → STATE = PAUSE

3. MODE_SELECT = HYBRID (default)

4. Run checklist (Page 21)

5. Wait for READY

6. If trading: ARM_MANUAL (only)

7. End session → PAUSE → logout


D5) Auto-mode ignition sequence (only when explicitly intended)

1. MODE_SELECT = AUTO

2. Verify all context gates pass

3. ARM_AUTO = ON (explicit)

4. Trigger must be exact

5. After execution: ARM_AUTO forced OFF, MODE may revert to HYBRID
