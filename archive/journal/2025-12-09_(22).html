PAGE 30 — Integration Overview


EchoAuth integrates through a three-layer architecture:


⸻


Layer 1 — Application Interface Layer (AIL)


The visible UI and interaction logic.


Responsibilities

• Render ASD-safe grids

• Capture user selections

• Display NI-AI-approved responses

• Maintain predictable timing and sensory settings


Inputs: touch, button selection, symbolic input

Outputs: UI state changes, confirmations, logs


⸻


Layer 2 — Core Runtime Layer (CRL)


The operational heart of EchoAuth.


Includes:

• ORI Kernel

• NI Interpreter

• UMCE Coherence Engine

• Safety Envelope

• Execution Layer


Responsibilities:

• Convert raw input → intent

• Validate intent for timing, logic, safety

• Execute only safe, approved actions


⸻


Layer 3 — Integration & Policy Layer (IPL)


External interfaces for school, home, and clinical environments.


Responsibilities:

• API communication

• Device trust anchoring

• Logging exports

• Admin & role policy application


⸻


Integration Workflow Summary

1. User selects an item on the grid.

2. AIL captures raw input and formats it.

3. CRL converts raw input → intent, checks coherence and safety.

4. Safe result returned to AIL.

5. UIL updates display, logs entries.

6. IPL handles device, caregiver override, or admin policy changes.


This flow is deterministic — the same input always produces the same result.

PAGE 31 — API Endpoints


These endpoints define how external systems communicate with EchoAuth.


⸻


SESSION MANAGEMENT


POST /session/start


Start a new trusted session.


Input:


{

  "device_id": "string",

  "user_id": "optional"

}


Output:


{

  "session_token": "abc123",

  "ori_signature": "vector"

}



⸻


INPUT PROCESSING


POST /input/submit


Submit grid selections or symbolic inputs.


Input:


{

  "session_token": "abc123",

  "input_payload": {

     "grid_index": 12,

     "object": "water",

     "text": null

  }

}


Output:


{

  "intent": "request_water",

  "safety": "clear",

  "confidence": 0.98

}



⸻


CAREGIVER OVERRIDE


POST /caregiver/override


Execute a safety-approved override.


Input:


{

  "session_token": "abc123",

  "pin": "****",

  "action": "approve_request"

}


Output:


{

  "status": "approved",

  "audit_id": "log_8893"

}



⸻


ACTIVITY LOG EXPORT


GET /logs/export


Exports non-PII clinical logs.


⸻


CONFIGURATION UPDATE


POST /config/update


Applies updated grids, sensory settings, or clinical profiles.


Output:

{ "version_id": "v2.1" }

PAGE 33 — Environment Configuration


EchoAuth requires a controlled, stable environment to ensure predictable behavior.


⸻


Deployment Profiles


Home Environment

• Device trust anchor

• Local-only mode

• Caregiver PIN required


School Environment

• MDM-managed device

• Sensory settings locked

• IEP-linked configurations


Clinical Environment

• Therapy-mode templates

• Extended logging

• Strict override controls


⸻


Required Configuration Files

1. ori_profile.json

Identity signature, user preferences.

2. safety_policy.yaml

System-wide safety rules.

3. sensory_profile.yaml

Pacing, color modes, animation settings.

4. environment_profile.yaml

Role permissions, override rules.


⸻


Developer Setup Requirements

• Node.js or Python runtime

• Local secure storage

• API keys for controlled modules

• Emulator or device for UI testing

PAGE 34 — Test Harness Instructions


This is the test suite required before any deployment.


⸻


1. Unit Tests


ORI Kernel

• Load identity

• Create new identity

• Reject invalid signature


NI Interpreter

• Correctly map inputs → intents

• Reject ambiguous intent


UMCE

• Block contradictory intent

• Enforce temporal constraints


⸻


2. Integration Tests


Input → Intent → UMCE → Safety → Output


Test cases must include:

• Fast repeated taps

• Sensory overload fail-safes

• Conflicting requests

• Emergency request detection


⸻


3. ASD-Specific Tests

• Predictability of UI

• Sensory stability under stress

• Override workflow latency

• Neutral fallback consistency


⸻


4. Caregiver Workflow Tests

• Override validity

• Lockout behavior

• Log export accuracy
