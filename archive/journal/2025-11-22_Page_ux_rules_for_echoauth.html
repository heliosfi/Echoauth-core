PAGE 7 — UX RULES FOR ECHOAUTH


Neuro-Safe, Sensory-Aware, NI–AI Guided User Experience Design


────────────────────────


These UX rules govern every visual, auditory, and interaction choice made inside EchoAuth.


They ensure the system always feels:

• predictable

• calm

• safe

• structured

• emotionally consistent

• ASD-friendly

• spiritually grounded (if chosen by the user)


These rules apply to ALL EchoAuth screens, components, characters, and flows.


⸻


⭐ SECTION 7.1 — CORE UX PRINCIPLES


These are the non-negotiable foundations that support ASD users and stabilize the nervous system.


1. Predictability & Pattern


Every screen must follow familiar patterns:

• same header

• same spacing

• same structure

• same tone

• same navigation logic


No sudden layout changes.


⸻


2. Low Cognitive Load


Information should be:

• chunked

• simplified

• sequential

• visually separated

• clean

• minimal


Avoid dense paragraphs or crowded screens.


⸻


3. Autonomy & Choice


ASD users relax when they have safe control.


Always provide:

• 2–3 choices

• clear options

• simple yes/no paths

• opt-out buttons

• “pause / breathe” options


Never trap the user in a forced flow.


⸻


4. Calm Color Palette


EchoAuth must use:

• soft blues

• muted purples

• gentle greens

• off-white grays


Avoid:

• neon

• flashing

• high contrast

• fast gradients


Color MUST align with sensory safety.


⸻


5. Minimal Animation


Animations may NOT be:

• sudden

• loud

• fast

• unpredictable


Allowed animations:

• slow fades

• soft breathing pulses

• very gentle transitions

• predictable character movements


Everything should feel like a calm ocean wave.


⸻


⭐ SECTION 7.2 — TEXT & LANGUAGE RULES


NI–AI governs ALL words and phrasing inside EchoAuth.


These rules must be applied to every prompt, button, and message.


1. Sentences must be short.


ASD-friendly processing works best with:

• 1–2 short sentences

• direct instructions

• no run-ons


⸻


2. Use literal language (no metaphor unless user can handle it).


Examples:

❌ “Let’s dive deep!”

✔ “Let’s look at this together.”


❌ “Time to climb the mountain!”

✔ “We can do this step by step.”


⸻


3. No sarcasm, teasing, or ambiguous tone.


Everything must be:

• clear

• direct

• kind


⸻


4. Use grounding language.


Examples:

• “You’re safe.”

• “I’m here with you.”

• “We can do this slowly.”

• “You don’t have to rush.”

• “Let’s take one step.”


Grounding phrases regulate the nervous system.


⸻


5. Ask permission before shifting subjects.


Examples:

• “Can I show you something next?”

• “Would you like another step?”

• “Is it okay if we continue?”


This gives the user control.


⸻


⭐ SECTION 7.3 — UI COMPONENT RULES


These are the exact rules for EchoAuth’s buttons, cards, and displays.


1. Button Size


Large, rounded, with:

• high readability

• soft colors

• minimal text (1–2 words)


Examples:

• “Start”

• “Pause”

• “Help”

• “Next”


⸻


2. Spacing


Generous spacing reduces stress:

• 24–32px around elements

• wide padding

• breathing room


Crowding = anxiety.


⸻


3. Icons & Characters


Icons should be:

• simple

• soft edges

• consistent style

• predictable animations


Characters (if used):

• repeat across sessions

• have stable emotional expression

• do not “act wild”


⸻


4. Visual Step Cards


All routines should use 3–5 step cards:


Card Structure:

1. Simple image

2. 3–5 words max

3. A “Done” checkbox


Examples:

• “Brush teeth → Done”

• “Put on shoes → Done”


This builds confidence and clarity.


⸻


⭐ SECTION 7.4 — INTERACTION RULES


1. Slow-to-Medium Pace


EchoAuth should never respond instantly if the user is stressed.


Use:

• soft delay (200–500ms)

• or a gentle breathing pulse


Fast responses can overstimulate.


⸻


2. Never ask multiple questions back-to-back.


One question → One answer → Gentle confirmation → Next step.


⸻


3. Always check emotional state before giving an instruction.


Examples:

• “Are you okay to continue?”

• “Do you want a shorter version?”

• “Want to pause?”


This prevents overload.


⸻


4. Routines MUST have clear beginnings and endings.


ASD users need closure.


EchoAuth must say:

• “We’re starting now.”

• “We’re done.”


⸻


5. Celebrate progress gently.


No loud excitement.

Use grounding praise:

• “Great job.”

• “You did that well.”

• “I’m proud of you.”

• “You stayed calm.”


⸻


⭐ SECTION 7.5 — SENSORY UX RULES (SPECIAL)


These apply to sound, vibration, and visuals.


1. No loud or high-pitched sounds.


Only:

• soft chimes

• rain sounds

• gentle tones

• warm pads

• low-frequency affirmations


⸻


2. Vibrations (if used)


Must be:

• short

• predictable

• optional


⸻


3. Safe Mode Visuals


When user is distressed:

• background dims

• brightness lowers

• blue filter activates

• UI shrinks to minimal


This is built from S12 + S11.


⸻


⭐ SECTION 7.6 — CAREGIVER UX RULES


Caregiver dashboards must:

• avoid clinical language

• avoid shame or negativity

• show growth, not deficits

• use color-coded progress (soft colors only)

• provide clear recommendations


Examples:

• “Today your child responded well to simple steps.”

• “Try shorter routines next time.”


NO:

• “Your child failed to follow directions.”

• “Your child struggled.”

• “Poor performance.”


Ever.


⸻


⭐ SECTION 7.7 — UMCE-GOVERNED UX CHECKS


Before ANY EchoAuth screen loads, UMCE must verify:


✔ Tone


✔ Emotional safety


✔ Sensory safety


✔ Cognitive load


✔ Field alignment


✔ Identity consistency


✔ ASD compatibility


If any fail → the UI switches to a safer, calmer mode automatically.


⸻


⭐ SECTION 7.8 — MVP VERSION OF UX RULES


For EchoAuth MVP (Page 5), include:

• soft UI theme

• calming welcome

• ASD-safe buttons

• predictable interaction patterns

• grounding language

• safe tone modes

• basic step cards

• slow pacing mode

• clear routines

• caregiver summary


These are the minimum to launch safely.
