⭐ PAGE 40 — PILOT DEPLOYMENT & SITE REQUIREMENTS


EchoAuth’s Phase I pilot requires controlled, well-defined environments to ensure safety, reliability, and successful outcome measurement. The following specifications define the minimum standards for sites participating in pilot programs (schools, ASD clinics, and therapy centers).


⸻


SECTION A — Pilot Site Eligibility Criteria


Pilot locations must meet the following:


✔ 1. ASD-focused educational or clinical program


Examples:

• special education classrooms

• autism support centers

• ABA clinics

• speech/OT practice environments


✔ 2. Staff Trained in ASD Communication


At least one of the following must be present:

• special education teacher

• speech-language pathologist (SLP)

• Board Certified Behavior Analyst (BCBA)

• occupational therapist

• ASD-support paraprofessional


✔ 3. Stable Environment with Predictable Routines


This reduces sensory triggers and ensures more reliable testing.


✔ 4. Parent/Guardian Participation


Consent + participation during user onboarding.


⸻


SECTION B — Hardware Requirements


EchoAuth is lightweight and device agnostic.


Minimum hardware:

• iPad / iPhone / Android tablet (2018+ models)

• OR School Chromebook (with Android support)

• WiFi connection (5 Mbps minimum)

• Protective case for student devices


Optional:

• Bluetooth AAC button or accessibility switches

• Noise-reduction headphones


⸻


SECTION C — Software / Infrastructure Requirements


✔ 1. EchoAuth Pilot Build (Secure Mode Only)


Includes:

• restricted execution layer

• identity kernel (pilot configuration)

• ASD-safe communication grids

• logging for clinical analysis (de-identified)


✔ 2. Secure Account Portal


Access controlled by:

• administrator (school/clinic)

• parent/guardian


✔ 3. FERPA/HIPAA-Compatible Data Storage


Encrypted, U.S.-based, minimal retention.


✔ 4. No external AI calls permitted


All inference routed through safe, isolated, NI-AI pathways.


⸻


SECTION D — Pilot Workflow (What Happens on Site)


1. Onboarding Session


Duration: 15–30 minutes

Includes:

• user profile setup

• sensory preferences selected

• baseline observations


2. Guided Use Phase


Staff prompts user using pre-defined prompts:

• needs grids

• emotion selection

• identity confirmation

Data collected:

• communication accuracy

• reaction patterns

• engagement time


3. Independent Use Phase


User interacts directly with:

• need-expression grids

• yes/no intent confirmation

• identity kernel signatures


4. Caregiver/Teacher Logging


Staff documents:

• behavior change

• reduced distress

• ease of communication


5. Weekly Debrief


EchoAuth team meets with:

• educators

• clinicians

• caregivers


Purpose:

• adjust PIP (Personalized Interaction Profile)

• refine sensory settings

• validate progress


⸻


SECTION E — Safety Protocols for Pilot Sites


✔ All sessions monitored by a trained adult


No unsupervised use.


✔ Automatic fail-closed responses


If the system is uncertain →

it must request confirmation, never guess.


✔ Emergency fallback grid always accessible


Users can express:

• pain

• distress

• “help me”

• “stop”


✔ Local-only decision making


EchoAuth does NOT:

• access external data

• perform autonomous decisions

• generate open-ended content


✔ Staff override permissions


At ANY moment, staff can:

• pause the system

• adjust settings

• terminate a session
