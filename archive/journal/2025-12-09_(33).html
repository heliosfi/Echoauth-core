📘 PAGE 45 — HARDWARE / SOFTWARE REQUIREMENTS


(SECTION VII — Prototype Roadmap)

────────────────────────────────────────


EchoAuth must operate reliably in school, clinic, and home environments with high sensory sensitivity, strict safety constraints, and variable technical capacity.

Accordingly, the system is designed to run on low-power, widely available devices while maintaining strict identity, safety, and coherence requirements.


This page defines:

• Minimum hardware specs

• Recommended hardware specs

• Supported operating systems

• Software dependencies

• Secure deployment requirements

• Networking and offline requirements


────────────────────────────────────────


1. Supported Device Types


EchoAuth is optimized for the following:


Primary Devices (ASD-Friendly Form Factor)

• Tablets (Preferred for ASD Accessibility)

• iPad (A10 chip or higher)

• Samsung Tab A (2019+) or equivalent

• Lenovo M10 series or equivalent


Secondary Devices

• Chromebooks

• Lightweight laptops

• Touchscreen Windows devices

• Dedicated assistive devices (future phase)


Why these devices?


✔ Lightweight

✔ Large touch targets

✔ Predictable, non-flickering displays

✔ Low sensory load

✔ High user-comfort among ASD children


────────────────────────────────────────


2. Minimum Hardware Specifications


To guarantee NI-AI stability, UMCE timing, and consistent sensory output:


CPU

• Quad-core 1.8 GHz (ARM or x86)

• Must support hardware encryption (AES)


Memory

• Minimum: 3 GB RAM

• Recommended: 4+ GB RAM


Storage

• 500MB free for core system

• 1GB additional for logs, user models, and updates


Display

• 60Hz refresh rate (NO adaptive/flicker-prone modes)

• Minimum resolution: 1280×720

• Touch latency < 100ms preferred


Connectivity

• WiFi (2.4 GHz acceptable)

• Bluetooth optional (for external devices)


Battery

• 6+ hours runtime (for school use)


────────────────────────────────────────


3. Recommended Hardware Specifications (Pilot-Level)


CPU

• Hexa-core or Octa-core ARM processors (A13 / Snapdragon 720G+)


Memory

• 4–6 GB RAM

• Ensures smooth NI Layer interpretation and UMCE cycles


Storage

• 2GB allocation for:

• User session logs

• ASD communication templates

• Offline modules


Additional Sensors (Optional but Ideal)

• Microphone for supervised emotion-intent tagging

(never open mic; only manual caregiver-triggered usage)

• Haptic motor (for sensory feedback modes — strictly optional)


────────────────────────────────────────


4. Supported Operating Systems


EchoAuth requires secure, sandboxed environments:


Mobile OS

• iOS 14+

• iPadOS 14+

• Android 10+


Desktop / Web OS

• ChromeOS 90+

• Windows 10+

• Ubuntu 20.04 LTS

• Browser support:

• Chrome, Edge, Firefox (latest versions)


Runtime Sandboxing


All deployments must meet:

• Encrypted storage

• Permission gating

• No background network exposure

• No unregulated OS-level AI assistants


────────────────────────────────────────


5. Required Software Dependencies


Core Languages & Frameworks

• Python 3.10 or NodeJS 18+ (backend)

• Flutter or React Native (UI layer)

• SQLite or sandboxed local DB (user/session storage)

• AES-256 & SHA-256 crypto libraries

• gRPC or REST for internal services


ML / NLP (STRICTLY LIMITED MODULES)


EchoAuth does not use open-ended LLMs.

All ML must be:

• Small

• Interpretable

• Contained

• Non-generative


Allowed frameworks (if needed):

• TensorFlow Lite

• ONNX Runtime

• Scikit-learn (for light classifiers)


No cloud-based LLMs are permitted in core functionality.


────────────────────────────────────────


6. Networking & Offline Requirements


EchoAuth is designed for offline-first ASD environments.


Offline Capabilities

• Full ORI Kernel

• NI Layer parsing

• UMCE validation

• Safety wrapper

• Communication grids

• Caregiver overrides

• Logging


Online (Optional) Features

• Data export

• Version updates

• School/clinic admin console sync

• Research-mode analytics (with consent)


Security

• All outbound connections must use TLS 1.3

• No unrestricted internet

• No third-party model calls

• No telemetry unless explicitly approved

• No cloud dependence for core functions


────────────────────────────────────────


7. Security Requirements


Device-Level

• Encrypted local storage

• Passcode or MDM restrictions

• Sandboxed app permissions


App-Level

• Safety invariants hardcoded

• No dynamic code loading

• No ability to install external plugins

• Caregiver-only configuration access


Deployment-Level

• Role-based authentication

• Log anonymization

• Optional no-network mode for high-sensitivity clinics
