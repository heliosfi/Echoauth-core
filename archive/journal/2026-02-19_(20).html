I. System Definition


EchoAuth is a deterministic runtime authorization gate for vulnerable care contexts.


It enforces authority resolution before execution.

Execution is mechanically impossible without a valid, fresh, cryptographically bound authorization grant.


The system separates:

• Identity Binding (non-executable)

• Authority Resolution (judgment)

• Execution (mechanical validation only)


Safety is achieved through structural constraints, not behavioral monitoring.


⸻


II. Architectural Separation of Powers


1. Identity Binding (UMCE)

• Establishes subject, actor, and role context.

• Produces signed, non-executable metadata.

• Cannot authorize or execute actions.


2. Management & Control Gate (MCG)

• Sole authority-resolution layer.

• Maps requests into:

• APPROVE

• DENY

• HOLD

• ESCALATE

• If APPROVE, issues AuthorizationGrant.


3. Control & Execution Gate (CEG)

• Performs no interpretation.

• Validates grant via deterministic 8-step contract.

• Executes only if all checks pass.


The system that identifies cannot authorize.

The system that authorizes cannot execute.

The system that executes cannot judge.


⸻


III. Authorization Grant Structure


A valid grant must include:

• grant_id (unique)

• issued_at_ms

• expires_at_ms

• root_authority_id

• actor_id

• envelope_id

• envelope_version

• authority_scope_id

• instruction_hash

• execution_context_hash

• state_lock_key

• sequence_nonce

• decision = APPROVE

• mcg_signature


The CEG rejects any grant missing required fields.


⸻


IV. Deterministic Validation Contract (CEG)


Execution requires successful completion of the following steps:

1. Schema & Decision Validation

2. Time Window Freshness Check

3. Signature Verification

4. Nonce Replay Check

5. Instruction & Context Hash Match

6. Lock Acquisition

7. Commit Point (Atomic Nonce Consumption)

8. Execution

9. Lock Release


Failure at any step prevents execution.


Failure classification:

• DENY → procedural invalidity

• HOLD → retry-safe environmental condition

• HALT → integrity or security violation


⸻


V. Runtime Invariants


1. Canonical Invariant


Payloads are canonicalized using RFC 8785 (JCS) prior to hashing.

Identical logical payloads produce identical hashes across languages.


2. Freshness Invariant


Expired grants are rejected.


3. Replay Invariant


Each grant is single-use.

Nonce consumed only at commit point.


4. Integrity Invariant


If instruction hash or context hash mismatches, execution halts.


5. Scope Invariant


Delegates cannot exceed parent-defined authority envelopes.


6. Concurrency Invariant


Exclusive state locks prevent simultaneous execution on same target.


7. Refusal-First Invariant


Authority channel loss defaults to HOLD, not execution.
