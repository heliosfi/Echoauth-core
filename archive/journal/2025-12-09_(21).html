PAGE 36 — ASD Sensory Safety Design


EchoAuth enforces strict sensory consistency to support autistic and non-verbal users.

These controls reduce overstimulation, promote emotional regulation, and improve communication accuracy.


⸻


1. Sensory-Stable Visual System


Color Palette

• Low-contrast blues, greens, and neutrals

• No bright red, flashing yellow, or unpredictable gradients


Movement

• Zero autoplay animations

• No dynamic UI shifts during interaction

• Transitions must be fade-only, 150–250ms


Layout Consistency

• Grid positions are fixed

• Icons never change location

• Predictability is enforced as a requirement, not a feature


⸻


2. Timing & Interaction Rules


Reaction Timing

• Minimum 300ms → Maximum 800ms

• Ensures consistency across devices and states


Lock-In Period

• Prevents double-taps or misfires

• Enforces a 1.2s window before another input is accepted


⸻


3. Cognitive Load Control


EchoAuth prevents overload by limiting:

• On-screen choices

• Text density

• Multi-step flows

• Auditory cues (none by default)


⸻


4. Stress-State Detection (Non-Physiological)


EchoAuth uses behavioral patterns, not biometrics:

• Fast repeated taps

• Aborted selections

• Grid-jumping behavior

• Timing irregularities


If detected → system shifts to Safe Mode:

• Simplified grid

• Larger icons

• Reduced options

• Caregiver alert option


⸻


5. Safeguard Against Harmful Stimuli


Non-permitted:

• Flashing content

• Loud audio

• Vibration feedback

• High-color saturation


These bans are permanent across all modes.

PAGE 37 — FERPA, HIPAA, COPPA Principles


EchoAuth is built to operate inside regulated educational, clinical, and pediatric environments.


This page defines the compliance spine.


⸻


1. FERPA Compliance (Schools)


EchoAuth does NOT store:

• Grades

• IEP documents

• Student academic performance


EchoAuth DOES store:

• Intent logs

• Session metadata

• Non-PII interaction summaries


FERPA Rules Enforced

• Data belongs to the school/district

• Parents may request log export or deletion

• No third-party sharing without written consent


⸻


2. HIPAA Compliance (Clinics + Therapy)


EchoAuth avoids handling PHI directly.


Protected:

• User ID is anonymized

• No medical diagnosis stored

• No clinician notes stored


Processing rules:

• All runtime data remains local unless encrypted and exported

• No cloud storage without explicit parent/guardian approval


⸻


3. COPPA Compliance (Children Under 13)


COPPA governs children’s online safety.


EchoAuth guarantees:

• No advertising

• No tracking cookies

• No location collection

• No personal profile building

• Parent/guardian PIN required for configuration changes


⸻


4. Cross-Environment Compliance Model


Environment Regulation Response

Home COPPA Local-only mode, parent PIN

School FERPA District-controlled device & logs

Clinic HIPAA Privacy-restricted runtime, no PHI


EchoAuth is engineered to fit all three without modifying core architecture.

PAGE 38 — Data Governance Model


EchoAuth uses a zero-waste, zero-exposure data philosophy:


“If the system does not need it, EchoAuth does not capture it.”


⸻


1. Data Categories


Category A — Operational Data


Required for real-time interactions:

• intent packets

• session tokens

• UI selections

• UMCE validation states


Purged automatically after session unless caregiver saves logs.


⸻


Category B — Clinical Logs (Non-PII)


Used for therapists or parents to observe patterns:

• timestamps

• event type

• intent classifications


These logs:

• are optional

• never include names

• can be fully deleted


⸻


Category C — Environmental Profiles


Defines:

• ASD sensory settings

• timing rules

• override permissions


Contains no identifying data.


⸻


2. Storage Rules

• Local-first storage; cloud is optional

• Logs encrypted with AES-256

• No sharing with third parties

• Strict version control on policy files


⸻


3. Retention & Deletion


Default retention:

• 0 days for operational data

• 30 days for logs (configurable)


Parent or school may:

• Export

• Delete

• Lock retention


⸻


4. Governance Enforcement Points


Governance is applied at 3 layers:

1. File/system level (encryption + access)

2. Runtime level (UMCE gating actions)

3. Policy level (role + environment rules)

PAGE 39 — Guardian-Override Channel


The override channel is a protected, non-negotiable safety mechanism.


It ensures caregivers or clinicians ALWAYS have final authority.


⸻


1. Override Requirements


All overrides require:

• PIN or passcode

• Active session token

• Approved device


Override expiration:

• 2 minutes or 1 action, whichever comes first


⸻


2. Override Actions


Examples:

• Approve restricted communication

• Unlock advanced grids

• Simplify UI

• Enter therapy mode

• Export logs


Overrides NEVER:

• Replace system safety rules

• Disable UMCE

• Allow unsafe output


⸻


3. Override Logging


Every override creates:


{

  "timestamp": "2025-12-08T18:55:00Z",

  "role": "caregiver",

  "action": "override_approve_request",

  "session": "abc123",

  "result": "success"

}


Logs cannot be hidden, modified, or suppressed.


⸻


4. Emergency Override (EO Mode)


Trigger:

• Repeated distress signals

• Elevated stress-state pattern


EO Mode:

• Reduces system to 3-button grid

• Calls caregiver mode

• Suspends complex logic pathways

PAGE 40 — Attack Surfaces + Mitigation


EchoAuth is intentionally non-networked by default to reduce attack exposure.


This section identifies possible risks and defines mitigation.


⸻


1. Local Device Compromise


Risk: Device-level malware or unauthorized access

Mitigation:

• No PII stored

• Role-based PIN

• Auto-lock after 60 seconds inactivity


⸻


2. Configuration Manipulation


Risk: Unapproved change to sensory or override settings

Mitigation:

• Settings locked behind caregiver PIN

• All config changes logged

• Version control integrity checks


⸻


3. API Abuse


Risk: Automated hits to endpoints

Mitigation:

• Rate limiting

• Session tokens with expiration

• ORI signature binding


⸻


4. Data Extraction


Risk: Logs extracted for misuse

Mitigation:

• AES-256 encryption

• Export requires override

• Non-PII logs only


⸻


5. Injection of Unsafe Outputs


Risk: Attempt to bypass NI-AI safety

Mitigation:

• UMCE mandatory approval

• Hard-coded invariants

• Fail-closed execution

PAGE 41 — Ethical Design Statements


EchoAuth follows six non-negotiable ethical principles.


⸻


1. User Autonomy


EchoAuth enhances autonomy for autistic users by enabling predictable, safe, and structured communication.


⸻


2. Non-Interference


EchoAuth does not:

• interpret emotions

• infer health data

• generate open-ended responses

• replace caregivers or clinicians


It supports — not replaces — human care.


⸻


3. Transparency


All actions:

• logged

• traceable

• explainable

• override-visible


No hidden processes.


⸻


4. Safety Over Accuracy


If the system must choose:

• Safety wins.

• Accuracy is secondary to harm prevention.


⸻


5. Minimal Data Philosophy


EchoAuth only uses what it needs:

• No biometric collection

• No location tracking

• No personal profiling


⸻


6. No Commercial Exploitation


EchoAuth will never:

• sell data

• use ads

• monetize logs

• build user psychological profiles


The system serves ASD individuals — not markets.
