You’ve got the right architecture here (risk gates + retest + time windows + news/event kill-switch), but the current script has several hard bugs that will break execution or silently corrupt results, plus a couple of “backtest validity” issues.


Below is a surgical review + patched version (still bar-based, still Polygon).


⸻


Critical bugs to fix (will break / misbehave)


1) targets =_atr_targets(...) is a NameError


You meant:


targets = calculate_atr_targets(entry, curr['atr'], state['pending_dir'])


2) is_power_hour(curr.name) is correct only because you set index to datetime


That’s fine, but be consistent: in some places you use curr.name, in others you built dt = curr['datetime']. You dropped datetime column and made it index, so use the index consistently:


dt = curr.name


3) Pending retest has no expiry


You defined RETEST_TIMEOUT_BARS, but your state doesn’t track pending_expiry_i or pending_confirm_i. Without expiry, the bot can fill a retest hours later (bad).


4) News filter is not backtest-compatible


In backtest, you’re calling Polygon “latest news” while iterating over 2020 candles. That will:

• disable trades randomly (based on today’s news),

• or rate-limit / slow the run.


Fix: only enable news filter in live mode, or mock news during backtest.


5) Your backtest dates are wrong for these levels


You’re still fetching 2020-09 while using 420/425 levels. Your own comment correctly flags this.


If you want this strategy validated, you must backtest on a period where TSLA is near those levels (e.g., Feb 2026). Otherwise results are nonsense.


6) Stop “pre-set” on retest can be inconsistent


You set state['stop'] = stop_base when arming the retest (good), but on retest fill you recompute targets using ATR of the retest candle, while stop uses ATR of the confirm candle. That mismatch is okay if intentional; if not, standardize: either

• compute both stop + targets off the confirm candle’s ATR, or

• compute both off the retest candle’s ATR.
