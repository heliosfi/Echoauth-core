ENGINEER HANDOFF PACK


MCG · NI-AI · EchoAuth


Document Purpose


This document defines what to build, what not to build, and how the systems interact at an implementation level.

It is intended for software engineers, system architects, and security reviewers.


⸻


1. SYSTEM BOUNDARIES (NON-NEGOTIABLE)


The system SHALL:

• Enforce execution only through explicit state alignment

• Treat non-action as a valid and successful outcome

• Maintain deterministic behavior at all times

• Preserve human authority without enabling unsafe override


The system SHALL NOT:

• Infer emotion, intent, or psychological state

• Predict outcomes or guarantee performance

• Escalate autonomously

• Allow execution by default


⸻


2. SYSTEM RESPONSIBILITIES (CLEAR OWNERSHIP)


2.1 NI-AI (Understanding & Explanation)


Responsibilities

• Organize inputs into structured context

• Explain system state and transitions to humans

• Maintain continuity across time


Explicitly NOT Responsible For

• Granting permission

• Triggering execution

• Changing MCG state


Implementation Notes

• Read-only access to MCG state

• Read-only access to EchoAuth authorization outcomes

• Output limited to explanation and summaries


⸻


2.2 MCG (Decision Discipline)


Responsibilities

• Maintain system state:

• OBSERVE

• CONFIRM

• EXECUTE

• Validate structural consistency

• Control state transitions


Explicitly NOT Responsible For

• Identity validation

• Permission enforcement

• Execution mechanics


Implementation Notes

• Deterministic state machine

• No external system can force state changes

• All state transitions must be logged


⸻


2.3 EchoAuth (Permission & Safety)


Responsibilities

• Authenticate identity

• Enforce role-based permissions

• Allow or deny execution strictly based on:

• Identity

• Role

• MCG state

• Authorization bounds


Explicitly NOT Responsible For

• Reasoning

• State determination

• Explanation


Implementation Notes

• Default state = DENY

• No cached execution permissions

• Enforcement on every execution attempt


⸻


3. INTERFACE CONTRACTS (LOCKED)


3.1 NI-AI → MCG


Structural Context Interface (SCI)


Payload

• Structured summaries

• Consistency indicators

• Context metadata


Restrictions

• No commands

• No urgency flags

• No recommendations


⸻


3.2 MCG → EchoAuth


State Authorization Interface (SAI)


Payload

• Current MCG state

• Execution bounds (if EXECUTE)


Restrictions

• No logic

• No interpretation

• No identity data


⸻


3.3 EchoAuth → NI-AI


Authorization Feedback Interface (AFI)


Payload

• Allow / Deny result

• Reason codes

• Boundary violations (if any)


Restrictions

• No secrets

• No enforcement logic

• No credentials


⸻


4. EXECUTION RULES (ABSOLUTE)


Execution is allowed ONLY IF:

1. MCG state = EXECUTE

2. EchoAuth authorization = ALLOW (BOUNDED)

3. Human authorization (if required) is valid

4. All boundaries are respected


If any condition fails → execution is denied.


⸻


5. FAILURE & ANOMALY IMPLEMENTATION


Failure Handling

• Immediate revocation of execution

• Forced return to OBSERVE

• Permission reset to DENY


Anomaly Handling

• No execution

• No escalation

• Logged as insufficient structure


Recovery is passive, not active.


⸻


6. HUMAN INTERACTION IMPLEMENTATION


Allowed Human Actions

• Observe system state

• Review explanations

• Authorize execution within bounds

• Revoke authorization

• Stop execution


Forbidden Human Actions

• Forcing EXECUTE

• Skipping CONFIRM

• Bypassing EchoAuth


All human actions must be:

• Authenticated

• Logged

• Auditable


⸻


7. LOGGING & AUDIT REQUIREMENTS


Every system MUST log:

• State transitions

• Authorization decisions

• Execution attempts

• Human interactions

• Failures and anomalies


Logs must be:

• Append-only

• Timestamped

• Tamper-resistant


⸻


8. SECURITY POSTURE (IMPLEMENTATION)

• Execution denied by default

• Separation of authority enforced in code

• No shared write access across systems

• No hidden control paths


Any system may halt execution.

No system may force execution.


⸻


9. ACCEPTANCE CRITERIA (ENGINEER CHECKLIST)


The system is considered correctly implemented if:

• Execution cannot occur without passing through all layers

• State skipping is impossible

• Non-action is logged as success when appropriate

• Human authority is bounded and auditable

• Failures reduce capability, never increase it


⸻


10. ENGINEER ROOM ONE-LINER


“NI-AI explains, MCG governs state, EchoAuth enforces permission—and execution only occurs when all three align.”
