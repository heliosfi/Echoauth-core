Title:

EchoAuth Local Launch Spec – Local Demo Runtime


1. Purpose


This document defines the local launch behavior for the EchoAuth MVP.

The goal is to start a small, self contained demo that shows:

• one caregiver account

• one child profile

• one check in request

• one output label from the NI AI engine


The local launcher is not for production use. It is for attorney review, grant reviewers, and engineers who need to see a concrete embodiment of the architecture.


⸻


2. Expected Files and Folders


The launcher assumes the following folders exist in the same project root:

• api/          – FastAPI or Flask HTTP endpoints

• core/         – NI AI logic: ORI, LRM, HER, PNRE, DAHE

• db/           – SQLite or simple JSON storage for demo data

• run_local_demo.py – main launcher script


Optional: a .env file for local config (port, log path).


⸻


3. Launch Order


When run_local_demo.py is executed, it must perform the following steps in order:

1. Load config

• read port, database path, and log file path

• fall back to safe defaults if missing

2. Initialize data layer

• create or open a local SQLite file (for example: echoauth_demo.db)

• create tables for: caregivers, children, checkins, nia_logs

3. Seed demo caregiver and child

• insert one caregiver row with a known email and password

• insert one child row linked to that caregiver

4. Start NI AI core

• load ORI identity module

• load LRM stress scoring module

• load HER runtime integrity modes (normal, safe, degraded)

• load PNRE singular output gate

• load DAHE append only history

5. Start HTTP server

• expose a small API on http://localhost:8000 or similar

• at minimum:

• POST /checkins/create – receive caregiver and child id and cues

• GET /health – simple health check

6. Wait for requests

• keep the process running until the user stops it with CTRL+C


⸻


4. Runtime Path for /checkins/create


When a caregiver sends a check in to /checkins/create, the local launcher must ensure the following order:

1. Bind identity (ORI)

• confirm that caregiver id and child id exist and are linked

• build an in memory identity capsule for this session

2. Score cues (LRM)

• map body cues and environment cues to a stress score from 0 to 5

3. Select mode (HER)

• if score is 0–1: mode = NORMAL

• if score is 2–3: mode = SAFE

• if score is 4–5: mode = DEGRADED

4. Resolve output (PNRE)

• NI AI engine may consider several internal labels

• PNRE must choose exactly one external label

• allowed labels in the MVP:

• SAFE

• NEEDS_ATTENTION

5. Store history (DAHE)

• write a row to checkins with:

• timestamp

• caregiver id

• child id

• stress score

• mode

• final label

6. Return response

• HTTP 200

• JSON body with:

• label (SAFE or NEEDS_ATTENTION)

• stress_score

• mode


No internal reasoning text, prompts, or conflict details are returned to the caregiver or child UI.


⸻


5. Safety Rules For Local Demo

• All logs stay on the local machine.

• No advanced UI elements, pop ups, or dark containers are required for the demo.

• Only one clear label per check in may be shown.

• No religious, doctrinal, or non scientific interpretation is included in the output.

• If the NI AI engine fails, the system must default to:

• log the error internally

• return NEEDS_ATTENTION with a generic safe message


⸻


6. Note For Attorney and Engineer


This local launch spec is an example embodiment of the EchoAuth architecture.

It shows how the following claimed subsystems can be wired together in code:

• ORI identity binding

• LRM stress score

• HER runtime integrity modes

• PNRE singular output gate

• DAHE history logging


The text of the patent claims is stored separately by the inventor and is not modified by this document.
