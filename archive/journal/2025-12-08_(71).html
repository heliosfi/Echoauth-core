📄 RFD-01 — ORI & Identity Subsystem


1. Purpose & Scope


The ORI & Identity Subsystem is the root of NI-AI.

It encodes, enforces, and protects:

• system identity

• tone & behavior

• allowed / forbidden states

• alignment constraints across all Floors


Nothing in NI-AI may run without ORI being active.


Scope:

• E1–E15, plus their connections into NI, Memory, Lattice, and UMCE.

• Used by every other subsystem via lightweight interfaces (no tight coupling).


⸻


2. Responsibilities (MUST DO)


ORI is responsible for:

1. Identity Vector Construction

• Encode an IdentityProfile into an internal vector representation.

2. Session Signature Generation

• Produce stable, unique signatures per session (session_signature).

3. Constraint Enforcement

• Maintain Core Alignment Constraints (CAC): allowed domains, prohibited themes, tone bans, etc.

• Maintain Forbidden State Register (FSR) and Allowed State Register (ASR).

4. Integrity & Drift Detection

• Run checks at boot and periodically to ensure identity wasn’t corrupted.

5. Safety Envelope

• Provide a wrapper around inputs/outputs (ORISafetyEnvelope) so no raw I/O bypasses identity rules.


⸻


3. Non-Responsibilities (MUST NOT DO)


ORI is not responsible for:

• Running any LLM / model inference.

• Interpreting semantics, emotion, or user intent (that’s NI / Interpretation).

• Planning actions or selecting tools (that’s Execution / Coherence).

• Storing conversation history (Memory subsystem handles this).


If ORI begins to “think” or “decide,” the design is wrong.


⸻


4. Inputs & Outputs


Inputs:

• IdentityProfile

• id, display_name, tone_style, forbidden_states, allowed_domains…

• user_traits

• descriptive fields used to build identity vector (e.g., “calm, technical, safe-first”).

• context

• boot context, environment info (e.g., deployment region, org, product).


Outputs:

• identity_vector

• numeric or structured representation used by other subsystems.

• session_signature

• opaque string used as a consistent key for memory + tracing.

• ori_enforced_input

• same as input, but validated and tagged by ORI.

• ori_metadata

• flags: identity_ok, constraints_enforced, integrity_status.


⸻


5. Key Data Structures


Use Pydantic (or similar) for typing.


class IdentityProfile(BaseModel):

    id: str

    display_name: str

    tone_style: str                # e.g. "calm-technical"

    forbidden_states: List[str]    # e.g. ["angry", "hostile"]

    allowed_domains: List[str]     # e.g. ["education", "support"]


class ORIState(BaseModel):

    identity_vector: Any

    session_signature: str

    integrity_ok: bool

    last_check_ts: float


These are returned / stored, not exposed directly to external clients.


⸻


6. Failure Modes & Safety Rules


Failure Modes:

1. Integrity Check Fails

• ORI cannot confirm its own configuration or vector state.

2. Constraint Violation

• Incoming input hits forbidden domain, tone, or state.

3. Drift Detected

• Identity vector diverges from baseline more than a configured threshold.


Required Behavior:

• On any failure:

• Fail closed:

• Block downstream processing.

• Return a standard safe refusal via UMCE (safe_refusal).

• Log:

• trace_id, session_id, failure_type, input_excerpt.


⸻


7. Implementation Notes / Tech Options

• Identity encoding can be:

• Static (lookup table) in early prototypes.

• Learned embedding later (but must remain immutable at runtime).

• Hashing:

• Use SHA-256 or stronger for session_signature.

• Configuration:

• All ORI settings come from config/identity_profile.yaml and config/safety_policies.yaml.

• No runtime mutation except via approved admin channels.


⸻


8. Testing & Acceptance Criteria


Engineers must provide tests that prove:

1. Boot & Integrity

• ORI can initialize with a given profile and pass integrity check.

2. Constraint Enforcement

• Inputs that violate FSR are rejected.

• Allowed domains pass through unchanged.

3. Session Stability

• Repeated calls with same context → same session_signature.

4. Fail-Closed Behavior

• Simulated corruption of identity vector → system refuses to process queries.


Once those tests pass, RFD-01 is “accepted.”
