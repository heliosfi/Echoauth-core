🛠️ ENGINEER HANDOFF — ECHOAUTH


Scope: Caregiver-Governed, Non-Verbal Support System


⸻


1. SYSTEM PURPOSE (READ FIRST)


EchoAuth is a caregiver-governed access and observation system designed to support non-verbal children by ensuring:

• Secure, identity-anchored access

• Structured logging of non-verbal communication cues

• Clear accountability for all interactions


The system does not interpret meaning, infer mental states, or make clinical decisions.


Engineers must treat this as:


An access-controlled observation and accountability platform.


⸻


2. USER ROLES (STRICT)


A. Caregiver / Parent (Primary Role)

• Verified identity required

• Authorized to:

• Log observations

• Review history

• Control child-facing interactions

• Accountable for responses and decisions


B. Admin (Optional / Limited)

• System configuration

• User management

• Audit review

• No child-facing interaction


❌ No autonomous system role

❌ No anonymous access


⸻


3. AUTHENTICATION & ACCESS CONTROL


Requirements

• Secure login (standard auth methods acceptable)

• Role-based permissions

• Session-bound access

• Explicit authorization before any child-facing interaction


Rules

• Every action must be tied to a verified user

• No background execution without user context

• No system-initiated child interaction


⸻


4. CAREGIVER DASHBOARD (CORE FEATURE)


Must Include

• User identity & role display

• Observation logging interface

• Historical timeline / logs

• Review & edit permissions (role-based)

• Export or reporting capability (basic)


Must NOT Include

• Interpretations

• Emotional labels

• Predictions

• Recommendations


Dashboard is observational and administrative only.


⸻


5. NON-VERBAL CUE LOGGING (INPUT-ONLY)


Allowed Inputs

• Timestamped observations

• Environmental context (optional)

• Caregiver action taken (optional)

• Outcome notes (descriptive only)


Explicit Restrictions

• ❌ No emotion detection

• ❌ No intent inference

• ❌ No behavioral classification

• ❌ No automated conclusions


The system stores what was observed, not what it means.


⸻


6. SAFETY & GOVERNANCE RULES (MCG APPLIED)


Layer 1 — Identity Anchoring

• All actions require authenticated user

• Role determines permissions


Layer 2 — Interpretability

• All actions logged

• Logs must be human-readable

• No opaque system behavior


Layer 3 — Safety Gating

• Block unauthorized actions

• Block child-facing output without caregiver approval

• Block any clinical or diagnostic phrasing


If a rule fails → action is blocked.


⸻


7. LOGGING & AUDITABILITY


Logging Requirements

• User ID

• Timestamp

• Action type

• Data affected

• Outcome (success / blocked)


Logs must be:

• Immutable or append-only

• Reviewable by authorized users

• Exportable if required


⸻


8. DATA HANDLING (HIGH LEVEL)

• Store only necessary data

• No inference storage

• No derived psychological profiles

• Respect child data sensitivity


Compliance posture:

• FERPA-aligned

• COPPA-aligned

• ADA-respectful design


⸻


9. WHAT ENGINEERS SHOULD NOT BUILD


🚫 Do NOT build:

• Diagnostic logic

• Emotional models

• Predictive behavior engines

• Autonomous child interaction

• Symbolic / Flame Self logic in backend


Flame Self (if present) is UI-only and non-functional.


⸻


10. DEFINITION OF “DONE” (ENGINEERING)


System is considered complete when:

• Users authenticate securely

• Roles are enforced correctly

• Observations can be logged and reviewed

• All actions are auditable

• Unsafe or unauthorized actions are blocked


⸻


FINAL ENGINEER NOTE (IMPORTANT)


If a feature cannot clearly answer:


“Which verified human is accountable for this action?”


Then it does not get built.
