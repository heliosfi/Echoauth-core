## What Course-Lock Means

Course-lock means the founder approval artifact path must advance only by
the next repo-supported governance step and must not drift into adjacent
work merely because adjacent work is visible.

Course-lock requires every path step to stay bounded to its stated purpose:

* a review file remains a review file
* an index entry remains traceability only
* a shell-format review remains shell-format review only
* a task-order review remains task-order review only
* a template-requirements review remains requirements review only
* a currentness review remains currentness review only
* a cleanup discussion remains separate from the active approval-artifact
  path

A founder approval path step is not complete unless it is committed,
pushed, indexed when required, and followed by a clean and synchronized
repo state.

## Source-Of-Truth Hierarchy

The course-lock source-of-truth order is:

1. The current explicit user task order.
2. The locked commit and current `git status` result.
3. The actual committed files under `docs/control-gates/`.
4. The `docs/control-gates/README.md` index, for traceability only.
5. The latest repo-supported next-lane inspection or checkpoint report.
6. Prior governance reviews, only where they are specific to the current
   path and do not conflict with a newer, narrower review.

README indexing is not approval. Commit history is not approval. Document
existence is not approval. Manual estimates are not repo-authored
authority.

## Required Sequence Discipline

The founder approval artifact path must preserve this discipline:

* inspect the current clean/synced repo state before acting
* create only the explicitly requested governance file when creation is
  authorized
* review and tighten only the explicitly named file when review is
  requested
* commit and push only the explicitly named file when commit is requested
* index the committed file in README only when indexing is requested
* commit and push only README when README index commit is requested
* run the requested validation after each step
* pause whenever the next safe lane is unclear

A next step cannot skip unresolved prerequisites, unindexed completed
lanes, dirty working state, missing validation, or ambiguous founder
approval language.

## What Prevents Circular Spec Bloat

Circular spec bloat is blocked by requiring each future lane to satisfy at
least one of these conditions:

* it is named by a current repo-supported future-work item
* it closes a specific prerequisite gap on the founder approval artifact
  path
* it indexes a completed lane in README
* it performs inspection-only next-lane analysis
* it performs separately authorized cleanup without interrupting the active
  path

A new review should not be created merely because another review can
describe it. If the next safe lane is unclear, the correct action is
inspection-only next-lane analysis, not implementation, artifact creation,
runtime work, broker work, trading work, deletion, or another speculative
review.
