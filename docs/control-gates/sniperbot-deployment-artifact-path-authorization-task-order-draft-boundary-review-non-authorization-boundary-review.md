## Non-Operative Draft Field Map

The following field map is a draft boundary for a possible future task
order. It is not a task order and cannot be used as one:

* Exact task title field: would identify the future task order title if a
  later founder-authorized task creates one.
* Exact target file path field: would identify the exact file path for the
  future task order if separately authorized.
* Exact purpose field: would state the bounded documentation/governance
  purpose and preserve non-execution limits.
* Exact scope field: would define the exact artifact-path authorization
  question and exclude artifact creation, approval records,
  implementation, deployment, runtime, trading, command, capital, and
  execution surfaces.
* Explicit non-execution boundary field: would state that the future task
  order is non-runtime, non-execution, non-trading, non-command, and
  non-capital-deployment unless later repo-supported authority explicitly
  says otherwise.
* Explicit exclusions field: would list founder approval, founder approval
  artifacts, artifact shells, artifact storage authority, artifact creation
  authority, approval records, approval activation, implementation and
  deployment authority, runtime authority, broker/trading authority,
  order-routing authority, command execution, paper/live execution, capital
  deployment, cleanup/deletion authority, and execution capability as
  excluded unless separately authorized.
* Traceability field: would cite the artifact-path authorization boundary
  review, task-order requirements boundary review, task-order review
  boundary review, founder-selected task-scope review, and course-lock
  review.
* Refusal-first language field: would require refusal when current,
  explicit, traceable authority is absent.
* Founder authority statement field: would identify the separate explicit
  founder authority for the future task order, if it exists.
* Validation requirements field: would require clean repo state, exact-file
  scope, no README edits unless separately requested, no staged extras, and
  `git diff --check` validation when applicable.
* Acceptance criteria field: would require exactly the authorized file
  state, no unapproved files, no runtime or execution effects, and no
  authority creation outside the exact task.
* Pause/escalation requirement field: would require pause and founder
  confirmation if authority is unclear, prerequisites are missing, repo
  content conflicts, or the path becomes authority-sensitive beyond the
  bounded task.

These fields are labels and boundary expectations only. They are not an
adopted template, not a usable task order, and not authority to create or
execute a task order.
