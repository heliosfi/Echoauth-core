Any future artifact-path authorization task order, if separately requested
and separately founder-authorized later, must include at minimum:

* separate explicit founder authority for the exact task
* exact task title
* exact target file path
* exact target scope
* exact non-execution boundary
* explicit exclusions for artifacts, approval records, implementation,
  deployment, runtime, broker access, trading access, order routing,
  command execution, capital deployment, paper/live execution, and
  execution capability
* traceability requirements tying the future task order to the current
  artifact-path authorization boundary review and founder-selected task
  scope review
* refusal-first language requiring denial when current, traceable,
  explicit authority is absent
* no founder approval artifact creation unless separately authorized
* no artifact shell creation unless separately authorized
* no artifact storage authority unless separately authorized
* no artifact creation authority unless separately authorized
* no approval record creation unless separately authorized
* no implementation or deployment task order unless separately authorized
* no runtime, trading, broker, order-routing, command, capital-deployment,
  paper/live execution, or execution effect

The future task order must remain bounded to the artifact-path question.
It must not combine artifact-path authorization with artifact creation,
approval record creation, founder approval, implementation, deployment,
runtime startup, broker connection, trading, order routing, command
execution, paper/live execution, capital deployment, or execution
capability.
