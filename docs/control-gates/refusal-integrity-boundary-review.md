## Core Rule

Refusal is a protective control, not a failure.

A refusal boundary must remain intact whenever authority, evidence,
delegation, monitoring, audit, or execution conditions are unclear.

## Refusal Integrity Principles

* Unsafe means refuse.
* Unknown means hold.
* Ambiguous means non-permission.
* Evidence is not authority.
* PASS is not permission.
* Silence is not consent.
* Documentation is not execution.
* Classification does not grant authority.
* Monitoring does not authorize action.
* Audit evidence does not authorize action.
* Parent/final authority cannot be transferred by evidence.
* Refusal must not be weakened by convenience, cached state, stale records,
  automation pressure, monitoring output, audit output, or broker/trading
  signals.

## Refusal Boundary Examples

Refusal integrity issues may include:

* Authority source is unclear.
* Evidence exists but authority is missing.
* Delegation scope is stale or incomplete.
* Parent instruction conflicts with caregiver/delegate evidence.
* Monitoring output suggests action but authority is unresolved.
* Audit record exists but does not prove permission.
* System state is incomplete, stale, or contradictory.
* Broker/trading signal exists but execution authority is absent.

These examples are documentation-review examples only. They do not become
approval, authority, execution permission, register mutation, blocker
resolution, event-bus progression, broker/trading permission, Robinhood
execution alignment, SniperBot behavior, macro/hotkey automation, or command
execution.
