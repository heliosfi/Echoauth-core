## Runtime Toggle / Workflow / Mechanism / Record Separation

Approval records, approval mechanisms, approval workflows, approval
runtime toggles, founder approval artifacts, implementation
authorizations, deployment authorizations, runtime startup
authorizations, broker authorizations, trading authorizations,
command-execution authorizations, and capital-deployment authorizations
are separate concepts.

An approval record is evidence of a decision. It may document a bounded
approval decision only if it is separately created, current,
task-specific, traceable, and explicitly authorized for the exact subject
it names.

An approval mechanism is a possible future evidence verifier only. If
separately authorized, it may check whether required evidence exists and
meets governance conditions. It cannot grant approval, create approval,
expand approval scope, or replace human review.

An approval workflow is a possible future evidence-handling and review
sequence only. If separately authorized, it may define how evidence is
received, reviewed, checked, rejected, recorded, or escalated for human
review. It cannot generate approval, replace an approval record, replace
an approval mechanism, create a founder approval artifact, authorize
implementation, authorize deployment, start runtime, connect brokers,
activate trading, execute commands, or deploy capital.

An approval runtime toggle is a possible future governed state boundary
only. If separately authorized, it may represent whether a bounded
approval state is blocked, pending, denied, expired, revoked, or
separately approved for a specific task. It cannot generate approval,
replace evidence, replace human review, create a founder approval
artifact, authorize implementation, authorize deployment, start runtime,
connect brokers, activate trading, execute commands, or deploy capital.

A founder approval artifact is a separate explicit artifact. It must be
created only under a separate explicit task order and must remain bounded
to the exact task, action, authority, scope, exclusions, evidence
reference, and currentness boundary it names.

Implementation authorization, deployment authorization, runtime startup
authorization, broker authorization, trading authorization,
command-execution authorization, and capital-deployment authorization
must each remain separate, explicit, current, scoped, and traceable.

None of these concepts creates authority by itself.

* a capital-allocation controller
