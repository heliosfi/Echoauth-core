## Purpose

The purpose of this review is to define the evidence package that would be
required before any possible future SniperBot advancement toward
live-money readiness could be considered.

This review exists because the SniperBot Live-Money Readiness Ladder
Non-Authorization Boundary Review defines staged future movement, and that
ladder requires evidence before any future advancement. This document
defines evidence requirement categories only. It does not create evidence,
accept evidence, validate evidence, certify readiness, approve advancement,
approve implementation, activate runtime, authorize paper trading,
authorize broker access, authorize order routing, authorize sizing,
authorize execution, authorize capital deployment, or authorize live-money
use.

## Evidence Requirements Overview

Evidence requirements define what future reviewers would need to see before
they could evaluate whether a later stage may be considered. They do not
prove that the evidence exists. They do not prove that the evidence is
current. They do not approve the evidence. They do not authorize
advancement.

Any future live-money readiness evidence package would need to separate at
least these evidence categories:

* governance document references
* approved task order references
* founder approval artifact references
* approval record references
* implementation boundary references
* simulation-only test evidence
* paper-trading evidence
* risk-control evidence
* stop/kill-switch evidence
* loss-limit evidence
* broker-boundary evidence
* order-routing-boundary evidence
* execution-prohibition evidence
* rollback evidence
* audit-log evidence
* trade-traceability evidence
* incident-review evidence
* human-review evidence

Listing an evidence category does not create that evidence category as an
artifact. Listing an evidence category does not accept, validate, certify,
or approve any evidence. Listing an evidence category does not authorize
simulation, paper trading, broker access, order routing, sizing,
implementation, production deployment, capital deployment, live execution,
or live-money activation.

## Evidence Required Before Simulation-Only Readiness

Before any possible future simulation-only readiness review could be
considered, a separate future evidence package would need to show:

* governance references to the live-money readiness ladder
* governance references to paper-trading / simulation boundaries
* evidence that simulation-only scope remains isolated from broker access
* evidence that simulation-only scope remains isolated from live accounts
* evidence that simulation-only scope remains isolated from paper accounts
* evidence that simulation-only scope remains isolated from order routing
* evidence that simulation-only scope remains isolated from position sizing
* evidence that simulation-only scope remains isolated from trade sizing
* evidence that simulation-only scope remains isolated from capital
  deployment
* evidence that no runtime toggles, broker credentials, API keys, order
  objects, order routes, execution adapters, or production deployment paths
  are created
* evidence that any future simulation task is bounded by a separate
  founder-approved task order, if such a task is ever requested

Simulation-only evidence requirements do not authorize simulation runtime
activation, replay runtime, backtest runtime, market-data ingestion, broker
sandbox access, paper trading, order routing, sizing, execution, or
readiness certification.
